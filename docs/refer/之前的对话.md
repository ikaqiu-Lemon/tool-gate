# 对话一

## 问题

Claude-mem有没有做工具治理？是否需要工具治理？能否将以下工具治理方式嵌入进去？

MySkills-Hub： Middleware 驱动的“渐进式披露 + 动态授权”Agent技能化框架项目描述： Agent存在“工具越多、模型越笨”的工程痛点，工具数量骤增易伴随上下文膨胀、tool 误选等问题。本项目设计并实现了一套基于Middleware的框架Skills-Hub，使得 Agent 按“先发现技能、再按需读取说明、后显式授权执行”的方式运行，使模型每轮仅获取与当前任务相关的最小工具集，兼顾可控性、可扩展性与复用性。技术栈：Python + LangChain + Middleware + Progressive Disclosure + Tool Gating + LangFuse 主要工作： 设计落地双平面架构：知识平面通过SkillPack + SkillIndexLoader + PromptComposer 实现技能索引、元数据缓存与渐进式披露；执行平面通过 SkillGate基于 allowed_tools 白名单执行授权，并通过重写 State 动态装配最小工具集，将工具装配升级为能力边界控制与执行治理，工具误选率降低80%； 基于Middleware实现Prompt/Tool动态重写：在 before_agent 阶段完成 Skills 元数据加载与缓存，在 wrap_model_call 阶段，通过PromptComposer实现无侵入式重写 prompt，注入技能目录供模型按需发现与读取；通过 request.override(tools=...) 仅下发当前任务所需的最小工具集合，降低全量的工具暴露带来的上下文噪声与 tool 误选概率； 实现会话级State缓存与回收控制：围绕多轮任务设计状态管理机制，维护 skills_metadata、skills_loaded、active_tools 等会话级状态；使用会话级 state 缓存技能元数据避免重复扫描与解析，对 read_skill 内容引入 LRU/TTL 缓存减少重复 IO，防止已启用的Skills持续堆积、工具集合再次膨胀，提升多轮对话下的稳定性与可控性； 基于 LangFuse 实现全链路观测与审计：围绕“发现技能→读取 SOP→显式授权→调用工具→完成任务”执行链路，以 trace / session / observation 串联 skill.read / skill.enable / tool.call / task.evaluate 关键节点，统一记录 token、latency、enable/read 命中率、工具误调用/重试/失败的原因分布，提升多轮任务下的排障效率、行为可解释性，便于分析与 Skills 优化方向。

# 回答

可以先下一个判断：

**Claude-mem 现在有“弱工具治理”，但还没有你说的那种完整的“渐进式披露 + 动态授权”工具治理。它已经在做的是“检索流程治理”和“失败降级治理”，还没有做到“按任务动态缩小工具面、按会话显式授权、按状态回收工具权限”这一级。** ([Claude-Mem](https://docs.claude-mem.ai/architecture/search-architecture?utm_source=chatgpt.com))

## 一、Claude-mem 现在有没有做工具治理

有，但比较轻，而且更像**使用协议约束**，不是完整的 runtime tool governance。最典型的例子是它的 memory search：文档把检索明确设计成 `search -> timeline -> get_observations` 的 3 层工作流，并用 `__IMPORTANT` 这个 MCP tool 把“先索引、再时间线、最后取全文”的规则直接教给模型；主干源码里也确实把这套流程编码成工具说明。这个思路本质上是在限制模型“怎么用工具”，属于一种治理，只是还停留在**工具使用路径**层，而不是**工具暴露边界**层。([Claude-Mem](https://docs.claude-mem.ai/architecture/search-architecture?utm_source=chatgpt.com))

它还有另一类治理：**主链路 fail-open**。hooks 架构明确强调“不能修改 Claude Code、本身不能拖慢主会话、失败不能把主会话搞挂”，而 `hook-command.ts` 也把 worker 不可用、超时、5xx、429 这类情况视为“graceful degradation”，直接放过主流程。这个是很成熟的工程治理，但治理的是**稳定性与可用性**，不是模型看到哪些工具、什么时候能调用哪些工具。([Claude-Mem](https://docs.claude-mem.ai/hooks-architecture?utm_source=chatgpt.com))

如果按你 MySkills-Hub 的标准来衡量，Claude-mem 目前**缺的部分**很明显：

- 没有 `allowed_tools` 这类会话级白名单机制。
- 没有 “read_skill / enable_skill / disable_skill” 这类显式授权动作。
- 没有把工具集按任务动态裁到最小。当前 MCP server 里工具列表是静态定义出来的。
- 没有围绕“技能已读 / 技能已启用 / 当前活跃工具 / TTL 回收”这类状态做专门的 session runtime。 ([GitHub](https://raw.githubusercontent.com/thedotmack/claude-mem/main/src/servers/mcp-server.ts))

还有一个很关键的背景：文档里曾经把它描述成带 `mem-search Skill` 的 skill-based 结构，但 issue #600 已经明确指出，主干里 `plugin/skills/` 不存在，搜索能力已经迁到 MCP tools，旧文档存在漂移。所以从“现在的主干实现”看，Claude-mem 不是一个 skills runtime，而是一个 **hooks + worker + MCP tools** 的系统。([Claude-Mem](https://docs.claude-mem.ai/architecture/overview?utm_source=chatgpt.com))

## 二、Claude-mem 是否需要工具治理

**需要，而且随着它继续长大，这件事会越来越需要。** ([Claude-Mem](https://docs.claude-mem.ai/architecture/search-architecture?utm_source=chatgpt.com))

如果它永远只有文档里那 4 个 memory MCP tools，治理压力其实还不算特别大，因为工具面很窄，`__IMPORTANT` 这种工作流约束已经能覆盖一部分误用风险。([Claude-Mem](https://docs.claude-mem.ai/architecture/search-architecture?utm_source=chatgpt.com))

但问题是，主干已经不止 4 个工具了。当前 `mcp-server.ts` 里除了 `search / timeline / get_observations / __IMPORTANT`，还已经有 `smart_search / smart_unfold / smart_outline` 这类代码导航工具。也就是说，它的工具面正在从“记忆检索”扩到“代码理解/探索”。这时候只靠一个固定的 3 层检索提示，就不够了。([GitHub](https://raw.githubusercontent.com/thedotmack/claude-mem/main/src/servers/mcp-server.ts))

再往后看，Claude-mem 还在往更多集成方向走。官方文档里已经有 OpenClaw integration，而且那里明确提到会在 `before_prompt_build` 阶段把 observation timeline 注入 system prompt。说明这个系统正在从“Claude Code 本地记忆插件”扩到“更通用的 agent memory substrate”。工具和场景一多，不做治理就会出现三个问题：
一是 prompt 里规则越来越多、上下文越来越噪；
二是模型在 memory tools 和 code exploration tools 之间容易误选；
三是不同模式、不同代理、不同任务对工具权限的需求明显不同。([Claude-Mem](https://docs.claude-mem.ai/openclaw-integration?utm_source=chatgpt.com))

所以，**Claude-mem 现在最缺的不是“再多几个工具”，而是把现有工具面做成可控的能力面。**

## 三、能不能把 MySkills-Hub 的工具治理方式嵌进去

**能，而且从工程上是很合适的。** 但要注意：**不是把你那套 Python/LangChain 代码原样塞进去，而是把它的治理思想迁移成 Claude-mem 现在这套 hooks + worker + MCP 的形态。** ([Claude-Mem](https://docs.claude-mem.ai/hooks-architecture?utm_source=chatgpt.com))

### 1）可以嵌，但嵌入点不能选错

Claude-mem 当前的核心链路是：

**hooks 采集 / 注入 → worker 持久化与处理 → MCP server 对模型暴露工具。** 文档和源码都说明了 MCP server 只是一个薄封装，真正的业务逻辑在 worker HTTP API 后面。([Claude-Mem](https://docs.claude-mem.ai/architecture/search-architecture?utm_source=chatgpt.com))

所以最合适的嵌入点不是数据库层，也不是单纯改 prompt，而是这两处：

第一处是 **MCP server 的工具暴露层**。
现在 `mcp-server.ts` 里是静态工具表。你完全可以把它改成“按 session / mode / active skill 动态返回工具列表”的形式。因为 MCP 的 `ListTools` 天然就是一个工具清单生成点，这里最适合做你说的 `request.override(tools=...)` 的等价实现。虽然 Claude-mem 当前源码里是静态 `tools` 数组，但这个位置正好就是可改造成动态装配的地方。([GitHub](https://raw.githubusercontent.com/thedotmack/claude-mem/main/src/servers/mcp-server.ts))

第二处是 **prompt/context 注入层**。
Claude-mem 已经有 `SessionStart` / `UserPromptSubmit` 的 context injection，也已经在 OpenClaw 集成里使用了 `before_prompt_build` 这类 prompt 装配点。这里非常适合接你的 `PromptComposer`：不是直接把全量 tool 说明塞给模型，而是先注入技能目录、当前已激活技能、若需更多能力请先读取/授权。([Claude-Mem](https://docs.claude-mem.ai/hooks-architecture?utm_source=chatgpt.com))

### 2）Claude-mem 需要的不是“恢复旧 skills 目录”，而是“在 MCP 之上加一层治理层”

这一点特别重要。issue #600 已经说明，旧的 `plugin/skills/` 路线在主干里不存在，文档里那套 skill-based 架构已经被 MCP tools 替换了。([GitHub](https://github.com/thedotmack/claude-mem/issues/600))

所以如果你要嵌 MySkills-Hub，**不要回退成老式 `plugin/skills/` 目录 + 一堆静态 SKILL.md**。更合适的做法是：

- 底层仍然保留 Claude-mem 的 MCP tools。
- 上面新增一层 **Skill Catalog / Tool Group / Authorization State**。
- 技能只是一个“工具组 + SOP + policy”的抽象，不直接替代 MCP。

也就是说，Claude-mem 的现实落法应该是：

**MCP 是执行面，Skills-Hub 是治理面。**

## 四、如果真要嵌，我建议你这样改

### A. 先定义“技能分组”，而不是直接治理单个工具

Claude-mem 现在的工具天然可以分成几组：

- `memory_recall`：`search`、`timeline`、`get_observations`
- `code_explore`：`smart_search`、`smart_unfold`、`smart_outline`
- `instructional`：`__IMPORTANT`

这一步很关键，因为你 MySkills-Hub 的价值不在“把 7 个工具改成 7 个开关”，而在于把它们提升成**可理解的能力边界**。([GitHub](https://raw.githubusercontent.com/thedotmack/claude-mem/main/src/servers/mcp-server.ts))

### B. 在 worker 里加会话级技能状态

Claude-mem 已经有本地 worker、SQLite、session 生命周期和 HTTP API，所以非常适合承载你那套状态：

- `skills_metadata`
- `skills_loaded`
- `active_skills`
- `active_tools`
- `tool_grants`
- `grant_scope = turn | session`
- `expires_at`

本地数据库本来就存在，而且 worker 本来就负责 session 与 observation 的处理，所以把 skill state 放在 worker 里比放在 prompt 或 hook 临时变量里稳得多。([Claude-Mem](https://docs.claude-mem.ai/architecture/database?utm_source=chatgpt.com))

### C. 把 PromptComposer 放到 context injection / before_prompt_build

你现有的 `PromptComposer` 思路非常适配 Claude-mem。
Claude-mem 自己已经会在会话开始时把近期上下文注入回来，OpenClaw integration 还会在 `before_prompt_build` 里注入 observation timeline。这个位置完全可以改成：

- 先注入 skill catalog 摘要
- 再注入当前 active skills
- 再注入当前允许的 tools
- 如果任务需要新能力，先调用 `read_skill` / `enable_skill`

这样就把 Claude-mem 原本“只注入历史记忆”的能力，升级成“历史记忆 + 能力边界提示”的组合。([Claude-Mem](https://docs.claude-mem.ai/hooks-architecture?utm_source=chatgpt.com))

### D. 把 SkillGate 放到 MCP server 分发前

Claude-mem 当前 MCP server 是“薄 HTTP 包装器”，而且工具分发点非常集中。这里最容易加一层：

- `beforeListTools(session)`：返回当前最小工具集
- `beforeCallTool(session, toolName, args)`：校验是否在 `allowed_tools`
- 不允许则返回“请先 read/enable 对应 skill”

因为现在的主干实现里工具面是从 MCP server 暴露出去的，所以这里就是最自然的 `ToolGate`。([GitHub](https://raw.githubusercontent.com/thedotmack/claude-mem/main/src/servers/mcp-server.ts))

### E. 加两个元工具，而不是一开始就让模型看到全部能力

这是你 MySkills-Hub 里最值钱的一部分，放到 Claude-mem 里也成立：

- `list_skills`
- `read_skill`
- `enable_skill`
- 可选 `disable_skill`

有了这几个元工具，Claude-mem 的行为就从“看到所有 MCP tools 后直接乱用”，变成：

**先发现可用能力 → 再读取 SOP → 再显式授权 → 再使用真实工具。**

这才是完整的渐进式披露。
现在的 `__IMPORTANT` 只是静态说明，而不是交互式授权流程。([GitHub](https://raw.githubusercontent.com/thedotmack/claude-mem/main/src/servers/mcp-server.ts))

## 五、哪些地方不能直接照搬 MySkills-Hub

这里也要说清楚：**能嵌，不代表能一字不改地嵌。**

你原来的 MySkills-Hub 是 `before_agent / wrap_model_call / request.override(tools=...)` 这种 LangChain / middleware 语义。Claude-mem 主干不是这个结构，它没有一个统一的 agent runtime middleware 栈；它更像是 **Claude Code host + hooks + sidecar worker + MCP server**。([Claude-Mem](https://docs.claude-mem.ai/hooks-architecture?utm_source=chatgpt.com))

所以要做语义映射：

- `before_agent`
  映射成 `SessionStart` / `UserPromptSubmit` / `before_prompt_build`
- `wrap_model_call`
  映射成 prompt 注入与 MCP `ListTools`
- `request.override(tools=...)`
  映射成 session-scoped 动态 `ListTools` 与 `CallTool` gate
- `state cache / TTL`
  映射成 worker 本地状态 + SQLite 持久化
- `LangFuse tracing`
  映射成 hook / worker / MCP 的统一 tracing

也就是说，你要迁的是**治理模式**，不是中间件 API 本身。

## 六、我对这个问题的最终结论

我的结论分三句：

**第一，Claude-mem 现在有治理，但是“弱治理”**：它治理了检索流程和系统可用性，还没有治理“每轮最小工具集”和“显式授权边界”。([Claude-Mem](https://docs.claude-mem.ai/architecture/search-architecture?utm_source=chatgpt.com))

**第二，Claude-mem 需要更强的工具治理**：尤其是在主干已经加入 `smart_search / smart_unfold / smart_outline`、并继续向 OpenClaw 等集成扩展之后。([GitHub](https://raw.githubusercontent.com/thedotmack/claude-mem/main/src/servers/mcp-server.ts))

**第三，MySkills-Hub 完全可以嵌进去**，但正确姿势不是“恢复老 skills 目录”，而是：
**在 Claude-mem 现有 MCP 执行面之上，加一层 session-aware 的 Skills governance layer。** ([GitHub](https://github.com/thedotmack/claude-mem/issues/600))

如果你愿意，我下一步可以直接给你出一版 **“Claude-mem × MySkills-Hub 融合架构图 + 模块改造点 + 工具状态机设计”**。

# 对话二

## 问题

全互联网搜索已经实现的完整的“渐进式披露 + 动态授权”工具治理的插件。即已经做到“按任务动态缩小工具面、按会话显式授权、按状态回收工具权限”这一级别的插件（任何开源插件、Skills或者MCP均可，不限制Claude Code，github优先）

## 回答

我全网搜完后的结论是：

**我没有找到一个“单仓、成熟、开源、直接把三件事都做满”的项目**——也就是同时做到

1. **按任务动态缩小工具面**，
2. **按会话显式授权**，
3. **按状态回收/撤销工具权限**。

但我找到了几类**非常接近**的实现，其中最值得你重点看的，是 **Janee、OpenClaw、Jido AI**。如果你要找“已经落地、不是论文概念”的开源参考，这三个最有价值。([GitHub](https://github.com/rsdouglas/janee?utm_source=chatgpt.com))

## 最接近“完整实现”的项目

### 1. Janee

这是我找到的**最接近你定义的单项目实现**。它不是通用 IDE 插件，而是一个 **MCP 权限代理/凭证代理**。Janee 把真实 API key/credential 藏在服务端，只向 agent 暴露“capability”。它有 `list_services` 做发现，`execute`/`exec` 做调用，`manage_credential` 做 **view / grant / revoke**；同时支持 `defaultAccess: restricted`、`allowedAgents`、`ttl`、`autoApprove`、`requiresReason`、请求级 `allow/deny rules`、`allowCommands`，并且 agent 运行时新建的 credential 默认是 `agent-only`，除非显式授权共享。它还把 session 做成 time-limited and revocable，并提供 `janee revoke <id>`。这已经非常接近你说的“渐进式披露 + 动态授权 + 状态回收”。([GitHub](https://github.com/rsdouglas/janee))

为什么我把它排第一：

- **渐进式披露**：先 `list_services` 发现能力和策略，再 `execute/exec` 真正调用，而不是把所有底层秘密和能力直接摊给模型。([GitHub](https://github.com/rsdouglas/janee))
- **显式授权**：`manage_credential` 能查看、授予、撤销 agent-scoped credential；`requiresReason` 和 `autoApprove` 也把授权策略显式化了。([GitHub](https://github.com/rsdouglas/janee))
- **状态回收**：`ttl`、time-limited sessions、`janee revoke`、agent-only 默认隔离，已经是标准的 JIT access / revocation 形态。([GitHub](https://github.com/rsdouglas/janee))

它和你目标的差一点在于：
它的“动态缩小工具面”更偏 **capability/agent/session-scoped**，不是一个通用 agent runtime 在**每一轮**都重写最小工具集的 middleware。但从工程效果上，它已经比大多数 MCP/技能系统更接近“真正的治理层”。([GitHub](https://github.com/rsdouglas/janee))

### 2. OpenClaw

如果 Janee 是“最像安全能力代理”的，那 **OpenClaw 是我找到的最像 runtime-level tool governance 的开源平台**。它已经有非常强的 **Gateway 级工具治理**：

- per-agent `tools.allow` / `tools.deny`
- tool profile / provider profile
- session 级 `/elevated on|ask|full|off`
- sender allowlist `tools.elevated.allowFrom.<provider>`
- fail-closed 的 channel/group allowlist
- session-level activation state（例如 WhatsApp 的 `/activation mention|always` 会更新 session state，而不是改全局配置）([OpenClaw](https://docs.openclaw.ai/concepts/delegate-architecture?utm_source=chatgpt.com))

它特别强的点是：
**工具边界不是写在 prompt 里，而是在 Gateway 层强制执行。** OpenClaw 文档直接写了 per-agent tool policy 是在 Gateway 层 enforce，独立于 agent personality；哪怕 agent 被 prompt 注入诱导去绕过规则，Gateway 也会拦截。([OpenClaw](https://docs.openclaw.ai/concepts/delegate-architecture?utm_source=chatgpt.com))

为什么它没有排第一：
因为我看到的是**强 runtime policy**，不是你说的那种完整 **“skill catalog → read_skill → enable_skill → active_tools 回收”** 路线。它更像“硬治理平台”，而不是“渐进式技能化治理平台”。也就是说，它很强，但重点在 **tool policy / sandbox / elevated / channel authorization**，不是技能发现式 progressive disclosure。([OpenClaw](https://docs.openclaw.ai/gateway/sandbox-vs-tool-policy-vs-elevated?utm_source=chatgpt.com))

### 3. Jido AI

Jido AI 是我找到的**最像你 MySkills-Hub 中 middleware 动态重写那一半**的开源框架。它的 README 直接写了：

- `allowed_tools` 可以对**单次请求**过滤 agent 的工具集
- `tools` 可以对**单次请求**替换整个 tool registry
- `request_transformer` 可以在每轮调用时做 **dynamic tool gating**
- 推荐让 tools 写 `StateOp.SetState`，再让 `request_transformer` 读 `context[:state]` 来收窄下一轮工具暴露
- 还有 checkpoint/resume 的 ReAct runtime。([GitHub](https://github.com/agentjido/jido_ai))

这已经非常接近你说的：
“按任务动态缩小工具面、按状态动态收敛下一轮工具暴露”。([GitHub](https://github.com/agentjido/jido_ai))

它为什么不是“完整答案”：
因为它更像 **runtime framework**，我没在它公开 README 里看到一套明确的 **用户显式授权 / grant / revoke / TTL reclaim** 交互机制。也就是说，它在“动态工具缩面”这件事上很强，但在“显式授权 + 状态撤销”这块没有 Janee/OpenClaw 那么成体系。([GitHub](https://github.com/agentjido/jido_ai))

## 做得很深，但只覆盖了一部分

### 4. ProDisco

ProDisco 是一个非常典型的 **progressive-disclosure MCP**：先 `searchTools`，再 `runSandbox`，并且每个 MCP client session 都有自己的隔离 Sandbox CRD、idle timeout cleanup 和 session limits。它很适合拿来研究“如何把能力暴露做成搜索优先、执行其次”的工具协议。([GitHub](https://github.com/harche/ProDisco))

但它的重点是：

- **渐进式披露**
- **隔离执行**
- **每 session 沙箱生命周期管理**

我没有在它公开 README 里看到与你要求等价的“显式授权 / grant / revoke 工具权限”的一层，所以它更像是 **progressive disclosure + isolated execution**，不是完整 dynamic authorization runtime。([GitHub](https://github.com/harche/ProDisco))

### 5. agent-secrets

`agent-secrets` 是我找到的**最强的 JIT credential governance 插件**之一。它既能作为 Claude Code/OpenCode 的 skill 使用，也有 OpenClaw plugin 形态；它提供 `secrets_lease`、`secrets_revoke`、`secrets_status`、`secrets_audit`、`secrets_killswitch`，还要求 lease 必须有 TTL，并提供 cleanup、后台 prune、append-only audit log。危险操作工具（如 `secrets_add` / `secrets_killswitch`）也需要显式 allowlist。([GitHub](https://github.com/joelhooks/agent-secrets))

这说明它在

- 显式授权
- TTL
- revoke
- 状态回收
  这几件事上做得很强。([GitHub](https://github.com/joelhooks/agent-secrets))

但它的治理对象主要是**credential access**，不是通用 agent tool surface。所以它更像“你方案里授权/回收层的一个专用组件”，不是完整的通用工具治理 runtime。([GitHub](https://github.com/joelhooks/agent-secrets))

### 6. Bitovi ai-coding-agent

这个项目也很有意思。它支持：

- prompt 绑定 `mcp_servers`
- 每个 MCP server 配置 `allowed_tools` / `allowed_paths`
- web dashboard 做 OAuth 授权
- 当 prompt 需要未授权 MCP 服务时，prompt 会被**排队**，用户完成授权后再执行
- 还提供 secure sessions、activity tracking、proxy status。([GitHub](https://github.com/bitovi/ai-coding-agent))

这已经实现了“任务级选择 MCP + 显式 OAuth 授权”。([GitHub](https://github.com/bitovi/ai-coding-agent))
但它自己也写了：OAuth tokens 目前是 **stored in memory only**，token refresh 还 **planned but not yet implemented**。所以它在“回收/撤销/长期状态治理”这块还不算完整。([GitHub](https://github.com/bitovi/ai-coding-agent))

## 只有“渐进式披露”，但还不算完整治理的项目

这一类我也找到了不少，最典型的是 **Koder、OpenHands software-agent-sdk、Conjure**。

- **Koder**：skills 只在启动时加载 metadata，需要时才 `get_skill(name)` 拉完整内容，并支持 `allowed_tools` 元数据；这是典型 progressive disclosure。([GitHub](https://github.com/feiskyer/koder/blob/main/docs/configuration.md))
- **OpenHands software-agent-sdk**：明确支持 AgentSkills 的 progressive disclosure，skill 会先出现在 `<available_skills>`，触发后再注入内容，也支持 `allowed_tools` 解析。([GitHub](https://github.com/OpenHands/software-agent-sdk/blob/main/openhands-sdk/openhands/sdk/context/skills/skill.py))
- **Conjure**：也是 metadata → body → resources 的 progressive disclosure，并且 native skills 可声明 `allowed_tools`。([GitHub](https://github.com/holsee/conjure))

但我在这些公开文档里**没有看到**像 Janee / OpenClaw 那样成体系的

- session grant/revoke
- TTL reclaim
- user-facing explicit authorization
  这类机制。它们更适合归类成“技能按需加载系统”，还不是“完整工具治理系统”。([GitHub](https://github.com/OpenHands/software-agent-sdk/blob/main/openhands-sdk/openhands/sdk/context/skills/skill.py))

## 我给你的最终排序

如果你的标准是你自己定义的那三级：

### 最接近完整实现

1. **Janee**：最接近单项目完成态。
2. **OpenClaw**：最强 runtime/policy/gateway 治理，但不是技能发现式。
3. **Jido AI**：最强动态缩面框架，但显式授权/撤销不完整。

### 很强但只覆盖一部分

1. **agent-secrets**：最强 JIT credential lease/revoke。
2. **Bitovi ai-coding-agent**：prompt 级服务选择 + OAuth 授权队列。
3. **ProDisco**：最纯粹的 progressive disclosure MCP。

### 只有 progressive disclosure，不算完整治理

1. **Koder / OpenHands SDK / Conjure**。([GitHub](https://github.com/rsdouglas/janee?utm_source=chatgpt.com))

## 一句话结论

**如果你要找一个“现成就能借鉴你 MySkills-Hub 思路”的开源基础，优先看 Janee；如果你要找一个“已经把 runtime 侧工具治理做得非常硬”的平台，优先看 OpenClaw；如果你要找“最像 middleware 动态缩面”的框架，优先看 Jido AI。**

你要的话，我下一步可以继续把这几个项目整理成一份 **“与 MySkills-Hub 的逐项对照表”**，直接按
`渐进式披露 / 动态授权 / 会话状态 / TTL回收 / 观测审计 / 是否适合二开` 六列给你排出来。

# 对话三

## 问题

如果我想要自己实现一个Claude Code的插件，能否将以上方法取长补短，然后实现我所想要的功能，要求是用Python实现，给出具体的实现方案和实现的Pipeline

## 回答

可以，而且**非常适合自己做一个 Claude Code 插件来落地**。Claude Code 官方插件本身就支持把 **skills、hooks、MCP servers** 打包成一个插件；插件提供的 MCP server 会在会话启动时自动连接，hooks 又能在 `SessionStart`、`UserPromptSubmit`、`PreToolUse`、`PermissionRequest`、`PostToolUse` 等生命周期点拦截和控制行为。再加上插件可以使用 `${CLAUDE_PLUGIN_DATA}` 保存持久状态，这正好够你实现一套“渐进式披露 + 动态授权 + 状态回收”的治理层。([Claude](https://code.claude.com/docs/en/plugins))

不过要先说一个关键现实：**不要把方案建立在“Claude Code 会稳定支持会话内动态刷新 MCP 工具列表”这个前提上。** MCP 规范本身支持 `tools/list` 和 `notifications/tools/list_changed`，服务器也可以声明 `listChanged` 能力；但 Claude Code 这边目前至少存在一个公开 issue，专门请求客户端在收到 `tools/list_changed` 后刷新工具列表。换句话说，规范层支持“动态加减工具”，但你在 Claude Code 里最好把它当成**未来增强项**，不是当前主方案。([模型上下文协议](https://modelcontextprotocol.io/specification/2024-11-05/server/tools))

所以，我建议你做的是一套**Claude Code 友好型 Python 插件架构**：

**外部看起来是 Claude Code 插件；内部是一个 Python 治理中台。**

------

## 一、建议你取长补短后的总思路

我建议你把几类项目的优点拼在一起：

- 借 **Jido AI** 的思路：按请求/按状态约束下一轮工具暴露，用 `allowed_tools`、`request_transformer` 那种“状态驱动的缩面”思想做你的决策内核。([GitHub](https://github.com/agentjido/jido_ai))
- 借 **OpenClaw** 的思路：做“基础 profile + allow/deny + 会话态 elevated/ask”的强治理，不把安全边界交给 prompt。OpenClaw 已经验证了 tool profile、allow/deny、session-only `/exec`、approval interlock 这套模式是可行的。([OpenClaw](https://docs.openclaw.ai/tools))
- 借 **Janee** 和 **agent-secrets** 的思路：高风险能力不要永久给，要做 **TTL、grant、revoke、audit**，任务完成或失败后及时回收。([GitHub](https://github.com/rsdouglas/janee))
- 借 **ProDisco** 的思路：把能力访问做成“先发现、再进入隔离执行”的两段式，不要一开始就把所有底层能力铺给模型。([GitHub](https://github.com/harche/ProDisco/blob/main/README.md))

这几者拼起来，正好就是你想要的三层能力：

1. **渐进式披露**：模型先看到技能目录，不先看到所有执行能力。
2. **动态授权**：需要时显式 enable，高风险走权限对话。
3. **状态回收**：TTL 到期、任务完成、任务失败都能 revoke。

------

## 二、最适合 Claude Code 的实现策略

因为 Claude Code 插件天然支持 **skills + hooks + MCP servers**，最稳的落地方法不是只做 skill，也不是只做 MCP，而是：

### 1. `skills/` 负责知识平面

只放：

- 技能目录
- 什么时候该用哪个技能
- 每个技能的用途、风险等级、示例
- “先 read_skill，再 enable_skill”的使用规则

技能本身不直接执行敏感操作，它只是“认知入口”。Claude Code 官方插件系统本来就支持 `skills/`，技能是插件根目录下的 `skills/<name>/SKILL.md`。([Claude](https://code.claude.com/docs/en/plugins-reference))

### 2. `.mcp.json` / `plugin.json` 里的 MCP server 负责执行平面

插件可以自带 MCP server，并且 Claude Code 会在 session 启动时自动连接。你用 Python 写一个本地 stdio MCP server 就可以。官方 MCP Python SDK 和 MCP 文档都支持这条路；MCP Python 侧要求 Python 3.10+、SDK 1.2.0+，而且 stdio 服务要把日志写到 stderr，不能污染 stdout 协议流。([Claude](https://code.claude.com/docs/en/mcp))

### 3. `hooks/hooks.json` 负责治理平面

Claude Code hooks 可以是 `command`、`http`、`prompt`、`agent` 四种。对你这个需求，我建议主用 **HTTP hooks**：Claude Code 会把事件 JSON POST 给本地服务端，你可以在 Python 里持久化状态、做授权判断、返回 allow/deny/ask 和 `additionalContext`。`PreToolUse` 可以 allow/deny/ask，还能改 `updatedInput`；`PermissionRequest` 可以代用户 allow/deny；`PostToolUseFailure` 能把失败上下文再喂回模型。([Claude](https://code.claude.com/docs/en/hooks))

## 三、我建议你做成这个架构

### 目标插件名

```
governed-tools`
或者
`skills-gate
```

### 核心原则

**Claude 看见的是“小而稳的能力面”；真实底层能力由 Python 治理层按状态放行。**

### 五层架构

#### 1. Skill Catalog 层

维护 `skills_metadata`：

- skill_id
- title
- summary
- risk_level
- required_grant
- allowed_ops
- default_ttl
- examples

这一层只负责“发现”和“解释”。

#### 2. Policy Engine 层

这是你的核心。
负责：

- 任务分类
- 计算候选 skills
- 计算 allowed_ops / allowed_tools
- grant / revoke / expire
- risk escalation
- 审计记录

#### 3. MCP Execution 层

只暴露少量 **元工具** 和少量 **低风险常驻工具**。
我建议先暴露这 6 个：

- `list_skills`
- `read_skill`
- `enable_skill`
- `disable_skill`
- `grant_status`
- `run_skill_action`

这样一开始 Claude 看到的工具面就很小，不会直接看到一堆危险具体工具。

#### 4. Hook Interceptor 层

用 hooks 强制执行：

- `SessionStart`：初始化 session / 清理过期 grant / 注入目录摘要
- `UserPromptSubmit`：任务分类 / 选择候选 skills / 注入提示
- `PreToolUse`：硬 gate
- `PermissionRequest`：高风险授权
- `PostToolUse`：更新状态 / 缩面
- `PostToolUseFailure`：失败回收 / 降权
- `SessionEnd`：清理临时授权

#### 5. State & Audit 层

用 SQLite 放在 `${CLAUDE_PLUGIN_DATA}` 下，持久化：

- sessions
- skills
- grants
- audit_logs
- task_runs
- revocations

Claude Code 官方文档已经明确插件可以用 `${CLAUDE_PLUGIN_DATA}` 保存持久状态，而且这个目录能跨插件更新保留。([Claude](https://code.claude.com/docs/en/mcp))

## 四、为什么我不建议你直接暴露所有具体工具

因为在 Claude Code 里，**真正“可靠”的动态缩面能力，当前更适合做成“固定小工具面 + 服务端分发子能力”**，而不是赌客户端会话内热刷新工具列表。

所以我建议你把“具体工具”藏在 `run_skill_action` 里面，让它按 skill 的 `allowed_ops` 去 dispatch，比如：

- `repo_read`: read / grep / glob / git diff
- `code_edit`: write / patch / format
- `verify`: test / lint / typecheck
- `secrets`: lease / status / revoke
- `web_ops`: fetch / search / crawl

这样做的好处是：

1. **Claude 表面上只看到 6 个工具**，天然缩面。
2. 真正的细粒度权限控制在 Python 服务端完成。
3. 即使 Claude Code 对 MCP `list_changed` 的支持不完整，你的治理也不受影响。

MCP 规范虽然支持动态 `tools/list` 和 `list_changed`，但鉴于 Claude Code 侧公开还有刷新工具列表的 enhancement 请求，这种“固定元工具 + 服务端子能力分发”的方案更稳。([模型上下文协议](https://modelcontextprotocol.io/specification/2024-11-05/server/tools))

## 六、具体实现 Pipeline

下面这条 pipeline，就是我建议你真正实现的主链路。

### Pipeline 0：插件加载

Claude Code 启动插件后，会按插件定义自动启动 MCP server。你的 `.mcp.json` 里把 Python 服务命令指到 `${CLAUDE_PLUGIN_ROOT}` 下的 Python 入口文件；状态文件写到 `${CLAUDE_PLUGIN_DATA}`。([Claude](https://code.claude.com/docs/en/mcp))

### Pipeline 1：SessionStart

`SessionStart` hook 触发后做三件事：

1. 检查 Python policy daemon 是否存活，不在就拉起。
2. 扫描 skill registry，加载 `skills_metadata` 到 SQLite / 内存。
3. 回收过期 grants，初始化本 session 的 `active_skills`、`risk_profile`。

然后通过 `additionalContext` 给 Claude 注入一段很短的目录摘要，比如：

- 当前只开放低风险能力
- 如需更强能力，请先 `list_skills -> read_skill -> enable_skill`

Claude Code 官方 hooks 支持在 `SessionStart` 等事件里通过 `additionalContext` 把字符串加进上下文。([Claude](https://code.claude.com/docs/en/hooks))

### Pipeline 2：UserPromptSubmit

用户发消息后，`UserPromptSubmit` hook 做任务分类：

- 这是阅读型任务还是改写型任务？
- 是否涉及 shell / network / secrets？
- 是否需要 elevated？

然后 policy engine 选出候选 skill 集合，例如：

- 只读：`repo-read`
- 编辑：`repo-read + code-edit`
- 验证：`verify`
- 涉密：`secrets`

这一步不直接开权限，只是把**候选目录**通过 `additionalContext` 告诉 Claude。
例如：

- 你当前适合使用 `repo-read`
- 如需编辑，请先读取 `code-edit` skill 并申请 enable

### Pipeline 3：Discovery

Claude 先调用：

1. `list_skills`
   返回 metadata 列表，只给：

- 名称
- 摘要
- 风险等级
- 是否已启用

1. `read_skill(skill_id)`
   返回 skill 的完整 SOP：

- 能做什么
- 需要什么权限
- 默认 TTL
- 可调用哪些 op
- 使用限制

这一步对应 ProDisco 的“先 searchTools 再执行”和你 MySkills-Hub 的“先发现、再按需读取”。([GitHub](https://github.com/harche/ProDisco/blob/main/README.md))

### Pipeline 4：Explicit Authorization

Claude 如果判断需要更强能力，就调用：

```
enable_skill(skill_id, reason, scope, ttl_requested)
```

然后 policy engine 走授权逻辑：

- 低风险：自动 grant
- 中风险：需要 `PermissionRequest`
- 高风险：必须 ask，必要时附带理由

Claude Code 的 `PermissionRequest` hook 可以在权限弹窗前介入，并 allow/deny。`PreToolUse` 也能返回 `ask`。([Claude](https://code.claude.com/docs/en/hooks))

这里建议你把授权设计成 Janee/agent-secrets 那种 lease：

- `grant_id`
- `skill_id`
- `scope = turn | session`
- `ttl`
- `granted_at`
- `expires_at`
- `status = active | expired | revoked`

### Pipeline 5：Execution

Claude 拿到 grant 后，通过 `run_skill_action` 执行具体操作：

```json
{
  "skill_id": "verify",
  "op": "run_tests",
  "args": {"target": "tests/auth"}
}
```

服务端先检查：

- skill 是否 active
- op 是否属于该 skill 的 allowed_ops
- ttl 是否未过期
- 风险级别是否需要额外 ask
- 参数是否越权

如果都过，就 dispatch 到底层 adapter。
例如：

- `repo-read` -> 文件/grep/git 只读适配器
- `code-edit` -> patch/format 适配器
- `verify` -> test/lint/typecheck 适配器
- `secrets` -> Janee 或 agent-secrets 适配器

这一步的关键是：**Claude 从头到尾都没拿到大而散的工具面，它只是在调一个治理过的执行入口。**

### Pipeline 6：PreToolUse / PermissionRequest 硬拦截

这一步是兜底硬治理。

Claude Code hooks 可以匹配 MCP 工具名；官方文档明确说 MCP 工具在 tool events 里会以普通工具出现，并遵循 `mcp__<server>__<tool>` 命名。`PreToolUse` 可以 deny / ask / allow，还能改输入。([Claude](https://code.claude.com/docs/en/hooks?utm_source=chatgpt.com))

所以即使模型因为 prompt 漂移直接乱调了：

- `mcp__governed-tools__run_skill_action`

你的 `PreToolUse` hook 也能拦截：

- 这个 skill 没 enable -> deny
- ttl 过期 -> deny
- 参数超范围 -> updatedInput 或 deny
- 高风险 -> ask

这一步借的是 OpenClaw 那种“tool policy 不交给模型记忆”的思路。([OpenClaw](https://docs.openclaw.ai/tools/exec-approvals))

### Pipeline 7：PostToolUse / PostToolUseFailure

工具成功后：

- 记录 token / latency / tool / skill / grant_id / result_size
- 更新 `last_used_at`
- 若是一次性权限，可自动降回只读 profile

工具失败后：

- 记录失败原因
- 对敏感 skill 可自动 revoke
- 用 `additionalContext` 把错误摘要喂回 Claude，引导它改走更安全路径

Claude Code 官方支持 `PostToolUseFailure.additionalContext`。([Claude](https://code.claude.com/docs/en/hooks))

### Pipeline 8：Stop / SessionEnd 回收

任务结束时做回收：

- turn-scope grant 全部 revoke
- session-scope 临时 grant 触发 TTL 检查
- secrets lease 自动回收
- 写 audit summary
- 清理临时文件 / 临时凭证

如果有“必须先释放才能结束”的危险能力，也可以在 `Stop` hook 上做强校验；Claude Code 的 hook 机制允许某些事件在 exit code 2 或 structured decision 下阻止动作继续。([Claude](https://code.claude.com/docs/en/hooks))

## 七、状态机怎么设计

我建议你把 skill 生命周期做成：

```
discovered -> loaded -> enabled -> active -> expired/revoked
```

再加一个高危分支：

```
enabled -> elevated_active -> revoked
```

每个 grant 至少要有：

```python
Grant(
    grant_id: str,
    session_id: str,
    skill_id: str,
    allowed_ops: list[str],
    scope: Literal["turn", "session"],
    ttl_seconds: int,
    status: Literal["active", "expired", "revoked"],
    granted_by: Literal["auto", "user", "policy"],
    reason: str | None,
    created_at: datetime,
    expires_at: datetime | None,
)
```

这样你就同时满足了：

- 按任务动态授权
- 按会话保存状态
- 按 TTL 回收
- 按失败/结束 revoke

# 对话四

## 问题

从使用场景上来分析，Claude Code有哪些需要进行工具治理的地方？注意是全套的工具治理

## 回答

要，而且**不只是 Bash 需要治理**。从 Claude Code 的真实使用方式看，它已经内置了一些零散控制：默认严格只读、`/permissions` 的 allow/ask/deny、`plan` / `acceptEdits` / `dontAsk` / `bypassPermissions` 等权限模式、hooks、subagent 工具白名单、MCP Tool Search、sandbox，以及项目/用户/托管三层设置。说明官方已经承认“工具面、权限、执行环境、团队设置”都需要控制；但这些还更像分散能力，离“全套工具治理”还有一段距离。([Claude](https://code.claude.com/docs/en/security))

如果按“使用场景”来拆，Claude Code 里**至少有 9 类地方必须做全套工具治理**。

## 1. 接手一个仓库、先读后想的场景

Claude Code 的内置工具里，`Read`、`Glob`、`Grep`、`LSP`、`ReadMcpResourceTool` 默认都不需要额外权限；而且 `additionalDirectories` 可以扩大 Claude 可访问的工作目录，subagent 默认还会继承主会话的全部工具，包括 MCP 工具。也就是说，“只读分析”看起来安全，但实际上已经涉及**代码、配置、密钥文件、外部资源**的可见范围治理。真正完整的治理不该只说“只读就行”，而要管：哪些目录能读、哪些路径永远禁止、哪些只读任务必须在 `plan` 模式下跑、哪些研究型 subagent 只能拿到 `Read/Grep/Glob/LSP` 这类最小读工具。([Claude](https://code.claude.com/docs/en/tools-reference))

## 2. 自动改代码、批量改文件的场景

`Edit`、`Write`、`NotebookEdit` 都是需要权限的改写工具；`Skill` 也可能触发带副作用的工作流。Claude Code 甚至专门说明：`bypassPermissions` 模式下虽然大多数提示会跳过，但对 `.git`、`.claude`、`.vscode`、`.idea` 这类目录仍会保留保护性确认。这个信号很明确：**文件修改不是一个统一风险等级**。所以全套治理至少要分层处理：普通源码、生成物、配置目录、CI/CD 文件、迁移脚本、密钥文件、IDE 配置、插件目录，都不该共用一条“允许写”规则。([Claude](https://code.claude.com/docs/en/tools-reference))

## 3. 跑命令、装依赖、测代码、改环境的场景

`Bash` 是 Claude Code 里最典型的高风险工具；官方默认要求权限，而且安全文档明确建议只有在受控环境里才考虑跳过权限提示。更关键的是，sandbox 不是“开了就万事大吉”：文档写得很清楚，sandbox 可以自动放行 Bash；`excludedCommands` 可以让某些命令在沙箱外运行；`allowUnsandboxedCommands` 甚至能决定是否彻底禁用 `dangerouslyDisableSandbox` 这个逃逸开关。再加上 Bash 的工作目录会在命令间保留、环境变量可通过 `CLAUDE_ENV_FILE` 或 `SessionStart` hook 注入，这就说明 Bash 治理必须是全套的：命令白名单、参数模式匹配、网络出口、环境变量注入、沙箱逃逸、超时、输出长度、工作目录漂移，全都要管。([Claude](https://code.claude.com/docs/en/tools-reference))

## 4. 联网搜索、抓网页、取外部上下文的场景

Claude Code 自带 `WebFetch` 和 `WebSearch`，两者都需要权限；同时权限规则又支持像 `WebFetch(domain:example.com)` 这样的域名级匹配。再结合 Bash 本身也能跑 `curl`、`wget`、`gh` 之类命令，你会发现“联网”其实有两条通路：**浏览器/网页工具通路**和 **Shell 网络通路**。这意味着完整治理不能只拦 `WebFetch`，还要把 Bash 网络命令一起纳入；否则你对网页工具设了域名白名单，Claude 仍可能改走 shell。([Claude](https://code.claude.com/docs/en/tools-reference))

## 5. 接大量 MCP 服务器、数据库、API、内部工具的场景

Claude Code 官方写得很直接：它可以通过 MCP 连接“数百个外部工具和数据源”。当 MCP 工具描述占上下文超过 10% 时，Claude Code 会自动启用 Tool Search，把工具改成按需发现、按需加载；而且 `/mcp` 能看到每个 server 的 token cost。另一方面，文档也提醒：MCP 连接可能在会话中途静默断开，工具会无提示消失。这说明 MCP 治理至少要覆盖三件事：
一是**工具面治理**，哪些 server 常驻、哪些 deferred、哪些只能手动 discover；
二是**数据面治理**，哪些资源能读、哪些 API 能调、单次 MCP 输出上限多少；
三是**连接生命周期治理**，断连、重连、失效、server 级禁用要怎么处理。Claude Code 甚至提供了 `MAX_MCP_OUTPUT_TOKENS` 和 `ENABLE_TOOL_SEARCH` 之类环境变量，说明这些已经是现实问题，不是抽象担忧。([Claude](https://code.claude.com/docs/en/mcp))

## 6. Skills、slash commands、plugins 自己也会变成“工具面”的场景

Claude Code 的技能不是纯文档。技能描述默认会在会话开始时加载，Claude 会按描述自动挑选技能；如果用 `disable-model-invocation: true`，技能才会在你手动触发前完全隐藏。官方还明确建议：**有副作用的技能应设成 `disable-model-invocation: true`**。另外，slash commands 里的 `!<command>` 会先执行 shell 命令，再把输出注入给 Claude。插件又可以把 skills、agents、hooks、MCP servers、LSP servers 一起打包。也就是说，skills / commands / plugins 本身就是需要治理的“上层工具面”：哪些技能可自动触发，哪些必须显式调用，哪些命令允许动态 shell 注入，哪些插件来源可信，哪些插件组件能进团队共享配置，这些都属于全套治理的一部分。([Claude](https://code.claude.com/docs/en/features-overview))

## 7. Subagent / Agent team 并行工作的场景

`Agent` 工具本身不需要权限，它会生成带独立上下文的 subagent；而 subagent 默认继承主会话的全部工具，包括 MCP。官方也给了显式例子：你可以用 `tools` 做 allowlist，只给 `Read/Grep/Glob/Bash`；也可以用 `disallowedTools` 去掉 `Write/Edit`。另外，hooks 在 subagent 里还能拿到 `agent_id` 和 `agent_type`。这说明**代理分工场景不是“主线程权限复制一下”那么简单**：研究型 subagent、审计型 subagent、执行型 subagent、外部集成型 subagent，工具面应该天然不同，而且授权、TTL、审计也应按 agent 维度拆开。([Claude](https://code.claude.com/docs/en/tools-reference))

## 8. 非交互、CI、远程 SSH、Desktop 远控的场景

CLI 支持 `--tools` 直接限制可用内置工具，也支持 `--permission-prompt-tool` 在非交互模式下把权限提示交给一个 MCP 工具处理。Desktop/SSH 场景下，Claude 是跑在远端机器上的，并且远程会话能继续使用权限模式、插件、MCP servers。Desktop 还允许启用 `bypassPermissions`，但企业管理员可以直接禁掉。这里最需要全套治理的，不是某个单独工具，而是**“宿主环境切换后的治理一致性”**：本机、容器、CI、远程开发机、跳板机，哪种环境允许哪种模式，谁能开 bypass，谁只能 plan，谁的权限只能 session 内生效。([Claude API Docs](https://docs.anthropic.com/en/docs/claude-code/cli-reference?utm_source=chatgpt.com))

## 9. 团队共享、组织托管、长期运行的场景

Claude Code 的设置有 managed、project、local 等作用域；文档明确说 project scope 很适合团队共享 permissions、hooks、MCP servers 和 plugins。管理员还能通过 managed settings 覆盖本地设置，甚至直接禁用 bypassPermissions。再加上 hooks 不只是观察工具，`PermissionRequest` 和 `PreToolUse` 还能改输入、改权限、改模式、增删目录，而且目标可以写进 session、localSettings、projectSettings、userSettings。说明在团队场景里，全套治理必须涵盖**谁能改规则、规则写到哪一层、是否允许本地覆盖、何时只在 session 生效、何时落盘**。否则工具治理很快就会从“安全控制”退化成“配置混乱”。([Claude](https://code.claude.com/docs/en/settings))

## 10. 审计、复盘、成本控制的场景

Claude Code 已经暴露了不少监控指标：`claude_code.code_edit_tool.decision`、token/cost、session 数、改动行数；在开启 `OTEL_LOG_TOOL_DETAILS=1` 时，还能记录 Bash 的命令字段、MCP 的 server/tool 名称、Skill 的 skill 名称。这说明官方已经把“工具使用可观测性”当成正式能力。完整治理因此不该停在 allow/deny，而要继续做：

- 谁启用了什么工具/技能
- 谁在什么会话里拿到过高权限
- 哪些 grant 被自动回收
- 哪些工具经常失败或被拒绝
- 哪些 MCP server 成本高、但几乎没带来有效结果。 ([Claude API Docs](https://docs.anthropic.com/en/docs/claude-code/monitoring-usage))

## 我对“Claude Code 全套工具治理”的归纳

如果把上面这些场景压缩成一套完整框架，Claude Code 真正需要的不是单点权限弹窗，而是这 8 层一起上：

1. **工具面治理**：哪些 built-in、哪些 MCP、哪些 skill、哪些 plugin 组件在当前任务里可见。Claude 已经有 Tool Search、技能自动加载和 `disable-model-invocation: true` 这种“部分缩面”能力，但还不够细。([Claude](https://code.claude.com/docs/en/mcp?utm_source=chatgpt.com))
2. **授权治理**：allow/ask/deny、permission mode、显式 enable、session grant、TTL、revoke。Claude 当前已有 `/permissions`、模式切换和 hook 更新权限的能力。([Claude](https://code.claude.com/docs/en/permissions))
3. **执行治理**：Bash sandbox、unsandboxed 禁用、命令/参数改写、超时、输出限制。Claude 当前已有 sandbox、MCP/Bash timeout、PostToolUse/Failure hooks。([Claude](https://code.claude.com/docs/en/sandboxing?utm_source=chatgpt.com))
4. **数据治理**：路径、目录、域名、资源 URI、MCP 输出、密钥文件、外部网页内容。Claude 当前已有路径/域名规则、目录增删、`MAX_MCP_OUTPUT_TOKENS`。([Claude](https://code.claude.com/docs/en/settings?utm_source=chatgpt.com))
5. **代理治理**：subagent/agent team 的独立工具面、隔离上下文和独立审计。Claude 当前已有 subagent allowlist/denylist 与 agent-specific hook context。([Claude API Docs](https://docs.anthropic.com/en/docs/claude-code/sub-agents))
6. **生命周期治理**：SessionStart、UserPromptSubmit、PreToolUse、PermissionRequest、PostToolUse、Stop、PostCompact 全链路收口。Claude hooks 已经覆盖这些关键节点。([Claude](https://code.claude.com/docs/en/hooks))
7. **组织治理**：managed/project/local 三层规则、团队共享插件/MCP/hook、管理员禁用 bypass。([Claude](https://code.claude.com/docs/en/settings))
8. **审计治理**：指标、日志、成本、决策、回收、异常复盘。([Claude API Docs](https://docs.anthropic.com/en/docs/claude-code/monitoring-usage))

所以，从使用场景上看，**Claude Code 几乎每个“能帮你提效”的地方都同时是“需要工具治理”的地方**：读仓库、改代码、跑命令、联网、接 MCP、开插件、起 subagent、跑远程、进团队。真正的全套治理，不该只问“这个工具要不要弹窗”，而要问：**当前任务为什么看到它、谁授权的、能用到什么程度、什么时候回收、出了事怎么追。**

你要的话，我下一步可以直接把这套分析收敛成一版 **“Claude Code 工具治理能力矩阵”**，按
`场景 → 风险 → Claude 原生能力 → 还缺什么治理能力` 四列给你排出来。

## 问题

所以，假设我在Claudecode中安装了很多的插件、MCP、Skills，我希望能够编写一个插件来对这些所有的工具、Skills进行治理，我具体要治理什么呢？给出一个系统性地解释，而不是单单讲述权限治理、渐进式披露节省Token和实验等等，先讲为什么做，再讲怎么做

## 回答

对，而且如果你的目标是：

**“我在 Claude Code 里装了很多 plugins、MCP、skills，希望再写一个治理插件，把这些能力统一管起来”**

那你要治理的，**绝对不只是权限**，也不只是“减少 token”或“少误调几个工具”。
更系统地说，你要治理的是：

**Claude Code 的“能力面”如何被看见、被理解、被授权、被组合、被执行、被回收。**

之所以要这么看，是因为 Claude Code 的扩展层本身就不只是“多几个工具”这么简单。官方文档明确说，Claude Code 可以通过 **plugins、MCP、skills、subagents、hooks** 扩展；而且这些特性加载时机不同：`CLAUDE.md` 和 MCP 会在会话开始时进入上下文，skills 先加载描述、调用时再加载完整内容，subagents 拿独立上下文，hooks 在外部确定性运行。也就是说，一旦你装很多插件、MCP、skills，Claude 面对的不是一个平面工具栏，而是一整套分层、异构、动态变化的能力系统。([code.claude.com](https://code.claude.com/docs/en/features-overview))

# 一、为什么要做“全套工具治理”

## 1. 因为你真正增加的不是“几个功能”，而是“能力面复杂度”

Claude Code 官方把 plugins 定义为可以打包 **skills、agents、hooks、MCP servers、LSP servers** 的自包含目录；MCP 本身又把 server capability 分成 **tools、resources、prompts** 三类。换句话说，你一旦多装几个插件，Claude 并不是只多了几个可执行函数，而是多了：

- 更多可执行动作
- 更多可读上下文
- 更多可复用 workflow / prompt
- 更多可委派代理
- 更多自动化 hook 入口

这会让系统从“一个助手 + 一组工具”变成“一个带扩展生态的 agent runtime”。这时候不治理，问题就不是单点误用，而是**系统整体失控**。([code.claude.com](https://code.claude.com/docs/en/plugins-reference); [modelcontextprotocol.io](https://modelcontextprotocol.io/specification/2025-11-25))

## 2. 因为 Claude Code 的扩展不是静态说明，而是会直接影响运行时行为

官方文档明确说：

- skills 会在相关时自动被 Claude 使用，或者手动触发；
- hooks 是确定性控制点，不是建议性提示；
- subagents 可以有独立的 tool restrictions、permission modes、hooks 和 skills；
- tools reference 里的工具名会被用在 permission rules、subagent tool lists、hook matchers 里。

这说明扩展层不是“参考文档”，而是**直接进入决策和执行链路**。
所以治理的目标，不该是“限制几个危险命令”，而是确保：

**Claude 在任何一轮里，看到的是恰当能力，做的是恰当动作，留下的是可解释状态。** ([code.claude.com](https://code.claude.com/docs/en/skills); [code.claude.com](https://code.claude.com/docs/en/hooks); [code.claude.com](https://code.claude.com/docs/en/sub-agents); [code.claude.com](https://code.claude.com/docs/en/tools-reference))

## 3. 因为多插件/MCP/skills 的真正问题是“能力冲突”和“边界漂移”

当你能力面很大时，最常见的问题不是“Claude 会不会调用某个工具”，而是下面这些更系统的问题：

- **看到太多**：模型一开始就面对太大的能力面，很难快速判断什么该用、什么不该用。
- **理解不清**：不同 MCP、plugin、skill 对相似任务提供重叠能力，但边界和副作用不清楚。
- **组合失控**：一个 skill 可能再触发工具，一个 subagent 可能再继承一堆工具和 skills。
- **状态漂移**：某次会话里打开的能力，下一轮是否还该保留，没有明确生命周期。
- **信任模糊**：到底哪些能力来自官方、哪些来自第三方插件、哪些来自本地 skill，很容易混在一起。
- **组织难控**：settings 支持用户级、仓库级等作用域，插件也能用户级或仓库级安装，不治理就会出现“每个 repo 一套规则、每个人一套口径”。([code.claude.com](https://code.claude.com/docs/en/permissions); [code.claude.com](https://code.claude.com/docs/en/settings); [code.claude.com](https://code.claude.com/docs/en/plugins))

所以，你做治理插件的本质目的应该是：

> 把“扩展生态带来的能力复杂度”收束成一套稳定、可解释、可控制的运行时能力边界。

------

# 二、你具体要治理什么

这里我不给你按“权限、token、实验”来拆，而是按**治理对象**和**治理维度**来拆。

------

## A. 先明确你要治理的对象

如果你在 Claude Code 里装了很多东西，真正要纳入治理台账的对象至少有 6 类。

### 1. 内置 Tools

比如 `Bash`、`Read`、`Edit`、`Write`、`WebFetch`、`WebSearch`、`Agent` 等。官方工具文档已经把这些列成标准工具，并明确哪些需要权限。([code.claude.com](https://code.claude.com/docs/en/tools-reference))

### 2. MCP Tools

MCP 规范里，tools 是 server 暴露给模型执行的函数。Claude Code 也明确支持 MCP servers，把外部服务接成可调用工具。([modelcontextprotocol.io](https://modelcontextprotocol.io/specification/2025-06-18/server/tools); [code.claude.com](https://code.claude.com/docs/en/mcp))

### 3. Skills

Claude Code 的 skills 会扩展 Claude 的能力，既可以是 reference skill，也可以是 action skill；它们会在相关时自动使用，或者被显式调用。([code.claude.com](https://code.claude.com/docs/en/skills); [code.claude.com](https://code.claude.com/docs/en/features-overview))

### 4. Subagents / Agents

Claude Code 的 `Agent` 工具会创建独立上下文的 subagent，而且 subagent 可以配置 tool restrictions、permission modes、hooks、skills。([code.claude.com](https://code.claude.com/docs/en/sub-agents))

### 5. Plugins

插件本身是能力容器，因为一个 plugin 可能同时带来 skills、hooks、MCP servers、agents。([code.claude.com](https://code.claude.com/docs/en/plugins-reference))

### 6. MCP Prompts / Resources

它们不是执行工具，但会改变模型“知道什么”和“怎么组织工作流”。MCP 规范里 resources 是上下文数据，prompts 是结构化消息模板。([modelcontextprotocol.io](https://modelcontextprotocol.io/specification/2025-06-18/server/resources); [modelcontextprotocol.io](https://modelcontextprotocol.io/specification/2025-06-18/server/prompts))

------

## B. 再明确你要治理的维度

真正的“全套治理”，不是只问“能不能用”，而是下面这 8 个问题都要有答案。

------

### 1. 可见性治理：**Claude 现在看到了什么能力**

这是第一层，也是最容易被忽略的一层。

你首先要治理的，不是授权，而是**暴露面**。

因为 Claude Code 的扩展层加载时机不同：MCP 和 CLAUDE.md 会在会话开始进入上下文，skills 先加载描述，subagent 拿独立上下文。([code.claude.com](https://code.claude.com/docs/en/features-overview))

所以你要管的是：

- 哪些 plugin 默认可见
- 哪些 MCP server 默认连上
- 哪些 skill 只显示目录摘要，哪些完整暴露
- 哪些 subagent 默认能被委派
- 哪些能力在当前任务根本不该出现

这层治理解决的是：

> “模型现在面对的能力面是不是太大、太乱、太早暴露了？”

这才是“渐进式披露”真正要解决的问题，但它只是全套治理的第一层，不是全部。

------

### 2. 语义治理：**Claude 如何理解这些能力**

仅仅看到能力还不够，还要治理“它怎么理解”。

因为技能、插件、MCP 工具的描述质量不同，边界可能重叠。
举例来说：

- 一个 skill 说“帮你部署服务”
- 一个 MCP tool 说“触发 CI/CD”
- 一个 slash command 也能做 deploy

如果你不治理语义层，Claude 会面对“多个看起来都像 deploy 的东西”，然后开始误选。

所以你要治理：

- 每个能力的用途定义
- 适用任务类型
- 副作用等级
- 与其他能力的边界
- 推荐调用顺序

也就是说，你不是只治理“能不能调”，而是治理：

> “Claude 应该把它理解成什么，不应该把它误解成什么。”

------

### 3. 资格治理：**在当前任务里，哪些能力有资格被使用**

这比权限更前一步。

权限是在问“允许吗”，资格是在问“连候选项都算不算”。

这层最适合按任务态来做，比如：

- 当前只是读仓库，不应出现写类能力
- 当前是 code review，不应出现 deploy 类 skill
- 当前只是总结信息，不应出现联网抓取和 shell 执行
- 当前 subagent 是 research 型，只能拿读类工具

Claude Code 已经支持 subagent tool restrictions、permission modes、hook matchers，这意味着宿主本身就有“分任务、分代理裁剪能力面”的接点。([code.claude.com](https://code.claude.com/docs/en/sub-agents); [code.claude.com](https://code.claude.com/docs/en/hooks))

这层治理的本质是：

> “不是所有已安装能力，都有资格进入当前任务的候选池。”

------

### 4. 授权治理：**谁批准了它现在可以执行**

这才是你说的“权限治理”，但它只是第四层。

Claude Code 官方已经支持 `/permissions` 的 allow / ask / deny，也支持 hooks 在 `PreToolUse`、`PermissionRequest` 等生命周期里拦截和返回决策。([code.claude.com](https://code.claude.com/docs/en/permissions); [code.claude.com](https://code.claude.com/docs/en/hooks))

但如果你要做全套治理，授权就不能只是一条 allow rule，而应该至少带上：

- 谁授的权
- 授给哪个对象（tool / skill / agent / plugin）
- 范围是本轮、当前 session，还是更长
- 是否有过期时间
- 是否能被自动撤销
- 是否要求理由
- 是否需要升级审批

所以授权治理真正要回答的是：

> “当前这次能力使用，合法性从哪里来，持续到什么时候，何时失效。”

------

### 5. 组合治理：**这些能力能不能彼此组合**

这是大多数人漏掉的一层，也是全套治理里非常重要的一层。

因为 Claude Code 的扩展不是单点调用，它会组合：

- skill → tool
- plugin → skill + hook + MCP
- agent → subagent → tools
- prompt / slash command → shell / MCP
- hook → 更新权限 / 增删目录 / 影响后续行为

官方文档明确说 hooks 是确定性控制点，subagents 能带自己的 tools/skills/hooks，plugin 又能打包这些组件。([code.claude.com](https://code.claude.com/docs/en/hooks); [code.claude.com](https://code.claude.com/docs/en/sub-agents); [code.claude.com](https://code.claude.com/docs/en/plugins-reference))

所以你必须治理：

- 哪些 skill 可以调用哪些工具
- 哪些 plugin 组件可以彼此组合
- 哪些 subagent 可继承哪些能力
- 哪些 action skill 不能再触发高风险 tool
- 哪些 MCP prompt 不能触发执行型工作流

这层治理解决的是：

> “单看每个能力都没问题，但拼起来会不会越权、失控或绕过边界？”

------

### 6. 生命周期治理：**能力什么时候生效，什么时候回收**

这是你特别关心的那一层，也是很多系统没有做全的地方。

真正完整的治理，不该只会“打开”，还要会“关闭”。

你要治理的生命周期包括：

- 安装后默认状态
- 会话开始时哪些能力自动出现
- 某个 skill 启用后持续多久
- 某个 subagent 的工具限制是否只在这个子任务内有效
- 某个 MCP server 断连/失效后如何撤销
- 某个 grant 到期后如何回收
- 会话结束后是否清理临时启用的能力

Claude Code 的 hooks 生命周期已经给了你很多控制点，例如 `SessionStart`、`PreToolUse`、`PostToolUse`、`Stop` 等。([code.claude.com](https://code.claude.com/docs/en/hooks))

所以这一层的核心问题是：

> “能力不是只有‘装了/没装’两种状态，而是有发现、启用、活跃、过期、撤销这些运行时状态。”

------

### 7. 信任治理：**这些能力是谁提供的，可信到什么程度**

如果你装了很多 plugin、MCP、skills，不同能力的信任级别显然不同：

- 官方内置工具
- 官方 marketplace 插件
- 团队内部插件
- 个人仓库技能
- 第三方 MCP server
- 临时试验插件

插件官方文档明确说，插件可以通过 marketplace 分发，并且支持用户级和仓库级配置。([code.claude.com](https://code.claude.com/docs/en/plugins); [code.claude.com](https://code.claude.com/docs/en/settings))

所以你需要治理：

- 来源归属
- 发布者身份
- 安装级别
- 是否经过组织批准
- 是否允许自动执行
- 默认风险等级

这层治理不是“技术执行”，而是“信任边界”。

------

### 8. 审计治理：**事后能不能讲清楚发生了什么**

最后一层不是控制，而是可解释。

如果你真要做一个治理插件，最终一定会被问：

- 为什么 Claude 用了这个 skill？
- 为什么这个 MCP 工具有资格出现？
- 为什么这个 subagent 拿到了这些 tools？
- 为什么这个权限在本轮过后被回收了？
- 哪些能力总被加载但几乎不用？
- 哪些 plugin 是噪声源？

Claude Code 已经有 hooks、permission rules、tool references 这些结构化接点，所以你的治理插件天然可以记录这些决策链。([code.claude.com](https://code.claude.com/docs/en/hooks); [code.claude.com](https://code.claude.com/docs/en/permissions))

这层真正要回答的是：

> “这套能力面为什么会在那一刻长成那样？”

------

# 三、所以你这个治理插件，核心不是“权限插件”，而是“能力运行时”

如果把上面所有内容收敛成一句话：

**你要治理的，不是某个工具能不能调用，而是 Claude Code 扩展生态下的“能力运行时”。**

这个运行时包含三件核心事：

## 1. 管暴露

让 Claude 在当前任务里只看到应该看到的能力。

## 2. 管编排

让 skill、tool、subagent、plugin 之间的组合有边界、有顺序、有资格。

## 3. 管状态

让能力的启用、活跃、过期、撤销都有明确生命周期。

------

# 四、最后给你一个最简洁的系统框架

如果你要把“具体治理什么”讲成一个面试或方案里的系统图，我建议你就讲这五个层次：

## 第一层：能力对象治理

管哪些对象进系统：

- built-in tools
- MCP tools
- skills
- subagents
- plugins
- MCP prompts/resources

## 第二层：能力暴露治理

管 Claude 现在看见什么：

- 默认可见
- 按任务可见
- 按代理可见
- 按会话阶段可见

## 第三层：能力执行治理

管现在能不能用：

- 资格判断
- 授权判断
- 参数/范围约束
- 组合约束

## 第四层：能力生命周期治理

管状态如何变化：

- discovered
- loaded
- enabled
- active
- expired
- revoked

## 第五层：能力审计治理

管事后是否可解释：

- 谁暴露的
- 谁授权的
- 谁使用的
- 何时回收的
- 为什么决策的

------

# 对话六

## 问题

给出你对工具治理插件的定位，是方便使用者进行工具治理（手动、提效），还是完全自动化的工具治理呢？

## 回答

我的定位是：

**它不应该是“完全自动化替你做决定”的黑盒，也不应该只是一个“手动点点点更方便”的配置面板。**
更合适的定位是：

**“以用户可控为前提、以自动化执行为主体”的半自动治理运行时。**

也可以说得更直接一点：

**决策权在人，执行权在系统。**

------

## 一、为什么我不主张一开始就做“完全自动化治理”

因为工具治理本质上不是纯技术优化问题，它里面混着三类判断：

### 1. 风险判断

比如这个 skill 现在该不该启用，这个 MCP server 该不该暴露，这个 subagent 能不能拿到写权限。
这里面很多时候不是“模型能不能判断”，而是“组织愿不愿意承担这个风险”。

### 2. 语义判断

同样是“修 bug”，有时候应该只读排查，有时候应该允许改代码，有时候还要跑测试。
这些不是静态规则就能完全吃掉的，尤其在复杂项目里，自动系统很容易把“相似任务”误当成“同类任务”。

### 3. 责任判断

一旦治理系统做了自动授权、自动放权、自动扩面，出了问题以后必须回答：

- 为什么当时自动给了这个能力？
- 谁定义的规则？
- 为什么没 ask？
- 为什么没回收？

如果一开始就走“完全自动化治理”，系统会很炫，但**不可信、不可解释、不可追责**。

所以我不建议把它定位成：

> “一个替用户自动管理所有工具权限的黑盒插件。”

------

## 二、为什么也不能只做“手动提效工具”

如果只做手动治理，比如：

- 帮用户列出已安装 plugins
- 帮用户开关 MCP
- 帮用户管理 skills
- 帮用户配 allow/deny

这种当然有价值，但它更像一个：
**配置中心 / 权限面板 / 工具管理助手**

这类产品的问题是，它解决的是“配置麻烦”，不是“运行时失控”。

而你想做的东西，其实不是静态管理台，而是：

- 当前任务下该看到什么能力
- 当前会话下谁有资格拿什么工具
- 当前 subagent 下哪些能力要收缩
- 某次授权何时自动失效
- 某个 skill 用完是否自动降权

这些都发生在**运行时**。
如果只做手动提效，真正关键的那一段就缺了。

所以它也不能只定位成：

> “帮助用户更方便管理工具配置的助手。”

------

## 三、我对它最合适的定位

我会把它定义成：

# **人控优先的自动化治理运行时**

这里面有三层含义。

### 第一层：用户始终拥有最高决策权

用户可以：

- 设定默认策略
- 决定哪些能力可自动启用
- 决定哪些能力必须 ask
- 决定哪些能力永远 deny
- 决定哪些授权只在本轮/session有效

也就是说，**治理边界是用户定的，不是插件替用户发明的**。

### 第二层：系统负责把规则自动执行到运行时

一旦用户定好了边界，系统就不该再让用户每一步都手动处理。
系统应该自动做：

- 按任务裁剪能力面
- 只展示候选 skill 目录
- 自动检查 grant 是否过期
- 自动阻断越权调用
- 自动回收临时授权
- 自动记录审计链路

也就是说，**运行中的重复动作应该自动化**。

### 第三层：高风险决策必须显式升级

自动化不是没有边界。
当出现下面几类情况时，系统应自动切回人工确认：

- 扩大能力面
- 从只读升到写
- 从本地升到外部系统
- 从普通工具升到 secrets / deploy / production 类工具
- 从主 agent 扩散到 subagent / multi-agent

也就是说，**低风险自动化，高风险显式授权**。

------

## 四、所以它不是“手动 or 自动”二选一，而是分层自动化

我会把它设计成三段式：

### 1. 策略定义阶段：偏手动

这一层由使用者决定：

- 默认工具面
- 风险分级
- 自动授权阈值
- TTL
- 回收规则
- 组织级/项目级/个人级策略

这一步强调“人控”。

### 2. 运行时治理阶段：偏自动

这一层系统自动完成：

- 能力发现
- 最小暴露
- 动态装配
- 授权校验
- 生命周期推进
- 过期回收
- 审计记录

这一步强调“自动执行”。

### 3. 升级审批阶段：偏人工

当运行时触发高风险边界时：

- ask
- require reason
- require approval
- 记录 grant 来源

这一步强调“可解释 + 可追责”。

------

## 五、如果只允许我给一句产品定位

我会这样说：

> **这个工具治理插件不是替用户夺走控制权的自动驾驶系统，而是一个“用户定义边界、系统自动执行治理、高风险动作显式升级”的治理运行时。**

------

## 六、如果你现在是做第一版 MVP，我建议的定位

第一版不要追求“完全自动治理”。

更稳的定位是：

### **默认手动授权 + 自动执行 + 自动回收**

具体来说：

- 能力面缩减：自动
- 常规低风险启用：自动
- 高风险 enable：手动确认
- grant 生命周期推进：自动
- TTL 到期回收：自动
- 审计记录：自动
- 策略修改：手动

这是最容易做出可信产品感的一版。

因为它既能体现你的工程设计：

- 不是纯 UI 配置器
- 也不是黑盒 AI 决策器

又能在面试里讲得很稳：

- 我把判断权和执行权分开
- 我把高风险决策留给人
- 我把重复治理动作交给系统
- 我把整个链路做成可解释状态机

------

## 七、我的最终判断

**定位上，我不支持“纯手动工具管理器”，也不支持“完全自动化黑盒治理器”。**
我支持的是：

# **可控的自动化治理**

也就是：

# **人定边界，系统执法，风险升级。**

这才最符合你前面一直在讲的那条主线：
**不是做一个权限面板，而是做一个真正的工具治理运行时。**

你要的话，我下一步可以继续帮你把这个定位收敛成一版 **产品定义文案 + 设计原则 + MVP边界**。
