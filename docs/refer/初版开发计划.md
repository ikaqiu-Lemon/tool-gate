# Claude Code 工具治理插件开发计划

## 一、项目背景与目标

### 1.1 背景问题

Claude Code 支持通过插件、MCP 工具及技能（Skills）扩展能力。随着外部工具、MCP 服务和技能数量的快速增长，代理模型在每轮推理中面对的候选能力急剧增加，出现上下文膨胀、模型误选和决策不稳定的问题。Skill‑Hub 项目总结的经验表明，单纯将所有工具一次性暴露给模型会显著降低工具选择的准确率。

本项目旨在针对 Claude Code 生态编写一个**工具治理插件**。插件通过 **渐进式披露** 和 **动态授权** 的方式在模型和工具之间增加一层运行时治理，以便：

1. **按任务动态缩减工具面**：在不同任务阶段只下发当前需要的最小工具集合，控制模型的候选集大小。
2. **显式授权**：用户或代理通过 `read_skill`/`enable_skill` 流程理解并显式启用技能，只有授权后才能使用对应工具。
3. **会话状态管理与回收**：维护技能元数据、已启用技能、当前可见工具等会话状态，通过 TTL/LRU 回收旧状态，避免会话越长越膨胀。
4. **符合 Claude Code 插件规范**：插件结构严格遵守官方指南，提供 `.claude-plugin/plugin.json` 清单，技能和钩子放置在正确的目录，并通过 hooks 和 MCP 配置接入运行时治理。

最终输出的插件应使用 **Python 语言 + LangChain 架构** 实现治理逻辑，并通过 Claude Code 的插件机制暴露。这项工作将为后续扩展到 OpenClaw、OpenCode 等平台打下基础。

## 二、整体架构概述

### 2.1 分层架构

根据 Skill‑Hub 项目的实践，治理逻辑应分成 **知识平面** 和 **执行平面** 两个平面，同时利用 Claude Code 插件的钩子系统控制模型的 prompt 和工具列表。在本项目中整体架构分为以下层次：

1. Python 核心治理层

   （governance core）

   - 使用 LangChain 编写，负责读取技能目录、判断任务类型、维护会话状态（`skills_metadata`、`skills_loaded`、`active_tools`）、执行授权策略、动态重写工具列表和 prompt 文本。核心逻辑不依赖具体宿主，便于未来复用。
   - 提供一个本地 HTTP/MCP 服务接口供插件调用，或者作为插件钩子的脚本被调用。

2. Claude Code 插件适配层

   - 按照官方插件规范在项目根目录创建 `.claude-plugin/plugin.json` manifest，并在根目录放置 `skills/`、`hooks/` 等目录。manifest 至少包含 `name`、`version` 和 `description` 字段。
   - 在 `hooks/hooks.json` 中配置事件处理，例如在 `SessionStart`、`UserPromptSubmit`、`PreToolUse`、`PermissionRequest` 等事件触发时调用 Python 治理层。Hooks 事件列表可从官方参考获取。
   - 如果需要暴露自定义技能，可以在 `skills/` 下编写 `SKILL.md`，并通过 `allowed_tools` 描述技能对应的可调用工具列表。

3. 技能目录与执行边界

   - 技能（Skill）不仅包含 SOP 描述、适用场景，还带有 `allowed_tools` 白名单。通过 `read_skill`/`enable_skill` 流程让模型先理解技能，再显式启用技能，将执行权限和知识披露分离。
   - 在执行平面，插件根据当前会话已经启用的技能集合重算出 `active_tools` 最小集合，再调用 Claude Code 的 tool schema 动态重写接口覆盖本轮工具列表。该机制参考 Skill‑Hub 中的动态重写策略。

4. 状态与缓存管理

   - 通过会话级状态保存技能目录 (`skills_metadata`)、已启用技能 (`skills_loaded`) 和当前可见工具 (`active_tools`)。每次 `before_agent` 钩子首先恢复状态，再基于需要更新。
   - 对技能说明文档使用进程内 LRU + TTL 缓存，减少重复读取，同时保证更新后的文档能及时生效。

5. 监控与审计

   （可选）

   - 与 Langfuse 结合对关键操作（skill.read、skill.enable、tool.call 等）打点，形成 trace / session / observation 的链路，可用于排障和性能分析。

### 2.2 数据流与控制流

下面描述一次典型的任务流程（简化版本）：

1. **会话初始化**（`SessionStart` 钩子）：恢复或初始化会话状态，构建技能索引，准备缓存。
2. **知识披露**（`wrap_model_call` 前）：注入技能目录摘要到 system prompt，并只提供基础入口工具，例如 `list_skills`、`read_skill`、`enable_skill`。
3. **用户/模型读取技能**：模型调用 `read_skill` 获取 SOP，用户确认需要启用某技能。
4. **显式授权**：模型调用 `enable_skill`；插件检查策略并更新 `skills_loaded`。此时根据技能的 `allowed_tools` 更新 `active_tools`。
5. **执行阶段**：根据重算后的 `active_tools` 调用实际工具。`PreToolUse` 钩子可再次检查权限，防止非白名单调用。
6. **收集观测与状态更新**：每次工具执行后更新状态；会话结束时将状态持久化到缓存（如 Redis），并写入审计日志。

### 阶段 2 ：核心治理逻辑实现

**目标**：在 Python 中实现运行时治理核心（governance core）。

1. **会话状态管理模块**：实现 `skills_metadata`、`skills_loaded`、`active_tools` 的状态对象；提供序列化与反序列化方法；支持基于 TTL/LRU 的缓存策略。
2. **技能索引构建与读取模块**：扫描指定目录读取技能元数据，解析 YAML frontmatter，建立索引供查询；实现 `read_skill(skill_id)` 方法按需读取技能内容并缓存。
3. **策略引擎**：根据当前任务（可通过模型提示或用户命令识别）判断需要启用的技能候选集合；实现 `enable_skill(skill_id)` 方法，检查权限策略并返回允许的工具白名单。
4. **工具重写器**：根据当前 `skills_loaded` 和每个技能的 `allowed_tools` 计算本轮 `active_tools`；提供接口返回动态工具列表，用于在 hooks 中替换 Claude Code 的工具 schema；同时输出需要注入的 system prompt 片段。
5. **接口服务**：可选：使用 FastAPI 实现一个 HTTP 服务，对外暴露 `read_skill`、`enable_skill`、`get_active_tools` 等 API，供插件钩子调用；亦可通过 `scripts/` 提供命令行脚本直接执行。
6. **单元测试**：为上述模块编写单元测试，确保索引构建、授权策略、缓存淘汰、重写逻辑正确无误。

**交付物**：

- `governance_core/` Python 包及测试代码
- 本阶段更新的技术文档（如 API 说明）

### 阶段 3 ：插件集成与钩子编排

**目标**：将核心治理逻辑封装成 Claude Code 插件，符合官方结构。

1. 插件目录初始化

   ：按官方建议创建插件目录，例如 

   ```
   tool-governance-plugin/
   ```

   ，其中包含：

   - `.claude-plugin/plugin.json`：定义插件的 `name`、`version`、`description`、`author` 等字段，可配置自定义 `hooks` 路径和 `mcpServers` 等。
   - `hooks/hooks.json`：配置需要拦截的事件，例如 `SessionStart`、`UserPromptSubmit`、`PreToolUse`、`PermissionRequest`、`PostToolUse` 等。每个钩子条目调用 Python 接口或脚本并根据返回结果调整行为。
   - `skills/`：若插件本身提供治理相关技能（如 `/tool-governance:list-skills`、`read-skill`、`enable-skill`），则在此目录编写 `SKILL.md`，在 frontmatter 中描述技能用途并声明 `allowed_tools` 白名单。
   - 可选 `bin/` 或 `scripts/`：存放调用 Python 核心的执行脚本，如 `run_read_skill.py`、`run_enable_skill.py`。Hooks 配置中可以通过 `${CLAUDE_PLUGIN_ROOT}/scripts/...` 调用它们。

2. 钩子实现

   ：

   - **SessionStart**：调用核心治理层初始化方法，加载或恢复会话状态。
   - **UserPromptSubmit**：在模型收到用户消息前识别任务类型，将技能目录摘要和入口工具注入 system prompt。
   - **PreToolUse / PermissionRequest**：在模型调用具体工具之前，询问 Python 核心是否允许当前调用；若工具不在 `active_tools` 中则返回 `deny` 或 `ask`；否则放行。
   - **PostToolUse / PostToolUseFailure**：工具执行后更新状态；必要时触发回收策略或记录观测数据。

3. **MCP 服务器配置（可选）**：若治理层以 HTTP 服务形式存在，可在 `.mcp.json` 中声明 MCP 服务器并定义工具，如 `list_skills`、`read_skill`、`enable_skill` 等，使其成为模型可以调用的外部工具。MCP 配置文件需要符合模型上下文协议（Model Context Protocol）规范，并应指定 `schema`, `methods`, `endpoint` 等字段。

4. **本地验证**：使用 `claude --plugin-dir <plugin-folder>` 加载插件进行功能测试，确保技能目录加载正确、hooks 生效、工具白名单能动态调整，并观察模型在任务过程中是否正确启用技能并限制工具集合。

**交付物**：

- 完整的 Claude Code 插件目录结构
- 通过本地测试的 hooks 配置和 Python 脚本
- 更新后的安装和使用说明

# Claude Code 工具治理插件：需求文档

## 1 项目目标

本项目旨在开发一个 Claude Code 插件，用于在代理模型调用外部工具时提供细粒度的治理能力。随着 Claude Code 支持的插件、MCP 工具和技能（Skills）越来越多，模型每轮都可能面对庞大的候选能力集，导致决策不稳定、误调用风险增加。本插件通过在模型与工具之间增加“渐进式披露+动态授权”的运行时治理层，降低上下文膨胀，提高工具使用准确率，保证安全性和可控性。

开发目标包括：

1. **按任务动态缩减工具面**：根据任务类型或模型提示动态计算当前会话允许暴露的最小工具集合，避免模型在无关工具中做选择。【Skill‑Hub项目】中的研究表明，不恰当的工具暴露会导致模型选择准确率下降，需要在运行时根据会话态收敛候选集。
2. **显式授权与知识披露**：通过 `list_skills`、`read_skill`、`enable_skill` 流程让模型先了解技能的 SOP 和适用场景，再显式启用技能并注入对应的工具白名单。将 “理解任务方法” 与 “执行权限” 分离，可以提高安全性并支持可追溯的治理链路。
3. **会话状态管理与回收**：维护技能元数据、已启用技能 (`skills_loaded`) 和当前可见工具 (`active_tools`) 等会话级状态。引入 TTL/LRU 缓存策略，防止长会话导致状态膨胀。会话结束后释放临时授权，确保状态一致性。
4. **插件化设计**：插件遵循 Claude Code 官方插件标准，将 manifest 文件放在 `.claude-plugin/plugin.json`，技能说明放在 `skills/` 目录，钩子配置放在 `hooks/` 目录，确保易于分发与部署。
5. **Python 实现与 LangChain 集成**：核心治理逻辑以 Python 编写，利用 LangChain 实现工具调用与 prompt 重写；插件层通过 hooks 调用 Python 核心模块，实现与宿主解耦，便于后续适配其他 IDE 平台。

## 2 用户需求

### 2.1 角色

1. **插件使用者**（AI 开发者或团队成员）：在 Claude Code 中执行编码或知识任务，需要可靠的工具组合，同时保证安全和效率。
2. **团队管理员**：配置插件策略，例如高风险工具需审批、默认 TTL、启用/禁用某些技能；关注审计记录与合规性。
3. **插件开发者**：维护治理逻辑和插件适配层，开发新策略或适配其他平台。

### 2.2 功能需求

1. **技能目录获取**：提供 `list_skills` 接口，列出所有可用技能的摘要，包括名称、简介、风险等级等，使模型能够发现新技能；目录信息由 Python 核心通过扫描 `skills/` 目录和元数据缓存提供。
2. **技能阅读**：提供 `read_skill(skill_id)` 接口，返回指定技能的 SOP、适用场景、推荐步骤、风险提示等，让模型先理解任务流程，再决定是否启用。【Skill‑Hub项目】提出先读后启用可降低误调用率。
3. **技能启用/禁用**：提供 `enable_skill(skill_id)` / `disable_skill(skill_id)` 接口，使模型显式申请启用或关闭技能。启用时更新会话状态中的 `skills_loaded` 并重新计算 `active_tools`；禁用时回收权限和工具。
4. **动态工具列表**：根据 `skills_loaded` 自动计算当前轮次的 `active_tools`，并通过 hooks 将此列表注入到 Claude Code 的工具定义中。若工具不在 `active_tools`，则拦截并返回授权提示。
5. **策略与权限管理**：允许管理员配置技能的默认风险等级、审批模式和 TTL；允许定义黑白名单或基于用户/项目的策略，决定是否能启用某技能。
6. **状态管理和缓存**：会话开始时初始化或恢复 `skills_metadata`、`skills_loaded` 和 `active_tools`；对技能元数据和 SOP 结果使用 TTL/LRU 缓存，减少重复读取；会话结束时持久化会话状态并回收临时授权。
7. **监控与审计（可选）**：记录关键操作（skill.read、skill.enable、tool.call 等）的 trace 和统计信息，支持分析误调用、耗时、授权请求频率等。

### 2.3 非功能需求

1. **性能要求**：治理层对每次工具调用增加的延迟必须可接受（建议 < 50 ms）。重写 prompt 和工具列表的开销要控制在合理范围。
2. **可扩展性**：架构解耦，Python 核心可独立于宿主存在；方便未来支持 OpenClaw、OpenCode 等其他编辑器。
3. **易于部署**：插件目录结构遵循 Claude Code 标准；`plugin.json` manifest 必须包含 `name`、`version`、`description` 等字段，其他配置项符合插件规范。
4. **安全与合规**：权限策略遵守团队的安全规范；审计日志完整、易于追溯；避免未经授权的命令执行。
5. **可维护性**：代码采用标准 Python 项目结构，使用类型提示和单元测试；文档完备，便于他人理解和二次开发。

## 4 风险与假设

# Claude Code 工具治理插件：技术方案文档

## 1 技术背景

Claude Code 提供了用于扩展 IDE 功能的插件系统。插件包含 manifest 和可选的技能目录、钩子、MCP 服务器等组件。官方文档要求 manifest (`.claude-plugin/plugin.json`) 位于 `.claude-plugin/` 目录，插件的技能和钩子放在根目录对应的 `skills/`、`hooks/` 目录。manifest 至少包含 `name`、`version`、`description`，可选字段包括 `author`、`keywords`、`hooks`、`skills`、`mcpServers` 等。钩子事件包括 `SessionStart`、`UserPromptSubmit`、`PreToolUse`、`PermissionRequest`、`PostToolUse` 等。

Skill‑Hub 项目提出通过 `read_skill` / `enable_skill` 将技能理解与执行授权分离，用 Middleware 在模型调用前拦截并重写 tools 和 prompt，动态收敛可见工具集合【Skill-Hub项目】。本技术方案结合上述思想与官方插件规范设计治理插件。

## 2 总体架构

插件分为三层：

1. 治理核心层

   （Python 包 

   ```
   governance_core
   ```

   ）

   - 实现技能索引构建、会话状态管理、授权策略评估、工具重写与 prompt 生成等核心逻辑。
   - 与宿主平台解耦，以便未来适配 OpenClaw、OpenCode 等其他 IDE。
   - 使用 LangChain 管理工具调用和 prompt 重写；使用 `cachetools` 实现 LRU/TTL 缓存；采用 `pydantic` 定义数据模型和配置。

2. 插件适配层

   - Claude Code 插件目录，包含 manifest、技能目录、钩子配置以及调度脚本。
   - manifest 指定插件名称、版本、描述、作者等基本信息和 hooks 列表；
   - 钩子文件 `hooks/hooks.json` 按事件绑定到 Python 调度脚本或 HTTP 服务；
   - 技能目录 `skills/` 用于声明治理相关技能，例如 `list_skills`、`read_skill`、`enable_skill`。

3. 工具/技能层

   - 由技能文件描述 SOP、适用范围和 `allowed_tools`；
   - 由 MCP 配置 (`.mcp.json`) 声明外部工具接口（可选），例如将核心接口包装为 MCP 工具；
   - 实际执行工具调用（如 `sql_query`、`search_docs`）由 LangChain 工具或 MCP 服务实现。

## 3 模块设计

### 3.1 治理核心层

该层封装在 `governance_core/` 目录下，主要模块如下：

1. `skill_indexer.py`

   - 扫描插件的 `skills/` 目录，读取每个 `SKILL.md` 的 YAML frontmatter，解析 `id`、`name`、`description`、`allowed_tools`、风险等级等元数据。
   - 构建 `SkillIndex` 数据结构，包含技能元数据和查找函数；通过版本号管理技能文档更新。
   - 暴露接口 `list_skills()` 返回简要目录； `read_skill(skill_id)` 读取并缓存完整内容。

2. `state_manager.py`

   - 定义会话状态对象 `SessionState`：包含 `skills_metadata`、`skills_loaded`、`active_tools`、`skill_read_cache`、`skill_enable_history` 等字段。
   - 提供序列化/反序列化方法，以 JSON 格式保存到缓存（如 Redis）或返回给 hooks。
   - 实现本地缓存（如 `cachetools.TTLCache`）存储技能元数据和读结果，减少磁盘或网络开销。

3. `policy_engine.py`

   - 根据技能配置、用户权限和策略定义判断是否允许启用某个技能：检查风险等级、是否需要审批、是否在白名单或黑名单中。
   - 管理授权生命周期：分成 `turn`、`session` 两种范围，支持 TTL（如默认 1 小时）和显式撤销。
   - 提供 `enable_skill(skill_id, session_state)` 和 `disable_skill(skill_id, session_state)` 接口，更新 `session_state.skills_loaded` 并返回授权结果。

4. `tool_rewriter.py`

   - 根据当前 `skills_loaded` 和每个技能的 `allowed_tools` 计算新一轮 `active_tools`。
   - 提供 `rewrite_tools(session_state)` 方法，返回可替换到 Claude Code 工具列表的 JSON 对象；可按任务阶段调整提示信息。
   - 生成系统提示片段，将技能目录摘要或 `allowed_tools` 描述插入 prompt，使模型理解当前可用能力。

5. `api_server.py`

   （可选）

   - 使用 FastAPI 构建一个轻量服务，提供 REST 接口：如 `/list_skills`、`/read_skill/{skill_id}`、`/enable_skill/{skill_id}`、`/disable_skill/{skill_id}`、`/get_active_tools`。
   - 钩子文件可以通过 HTTP 调用这些接口获取数据，实现插件与 Python 核心的松耦合。

## 4 关键流程与算法

### 4.1 技能读取与授权流程

1. 模型或用户调用 `list_skills`：`governance_core.skill_indexer.list_skills()` 返回技能目录摘要（名称、描述、风险等级）。
2. 调用 `read_skill(skill_id)`：`skill_indexer.read_skill()` 读取指定技能的 SOP 文档，并在需要时从缓存加载。返回适用场景、步骤、注意事项、`allowed_tools` 列表等。
3. 调用 `enable_skill(skill_id)`：`policy_engine.enable_skill()` 首先检查配置和策略（风险等级、审批需求、TTL），若通过则将 `skill_id` 写入 `state.skills_loaded`，并返回授权成功。授权结果包括生效范围（turn/session）、过期时间、允许使用的工具列表等。
4. 调用 `disable_skill(skill_id)`：移除 `state.skills_loaded` 中对应的技能，并从 `active_tools` 中去除该技能的工具列表。如果启用时设置了 TTL，治理层也会在到期后自动执行撤销。

### 4.2 工具重写逻辑

在每个 `SessionStart`/`UserPromptSubmit`/`PreToolUse` 钩子调用中，执行如下流程：

1. **恢复状态**：调用 `state_manager.load_state(session_id)`，若存在则恢复会话；否则初始化默认状态。
2. **判断任务类型**：可通过读取当前 prompt 或用户命令，结合技能 metadata 判断任务归属的技能候选集合。
3. **计算 `active_tools`**：遍历 `state.skills_loaded`，获取各技能的 `allowed_tools` 并求并集，再过滤掉未安装的工具或管理员禁用的工具。
4. **构造工具列表**：生成工具定义 JSON，用于覆盖 Claude Code 默认的工具集合。工具列表中可包含自定义治理工具，例如 `list_skills`、`read_skill`、`enable_skill`。
5. **注入系统提示**：在返回结果中附带建议的 system prompt 片段，例如技能目录摘要或权限提示，引导模型走 `read_skill`→`enable_skill` 的流程。
6. **更新状态并持久化**：调用 `state_manager.save_state()` 将最新状态写回缓存；必要时更新时间戳以支持 TTL。

若工具调用不在 `active_tools` 中，钩子返回 `deny` 或 `ask` 以阻断调用；对于高风险工具，钩子可以要求人工审批或返回提示信息。
