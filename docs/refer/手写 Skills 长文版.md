# 架构
我当时做这个项目的起点很朴素：我想把 **Skills** 这套东西真正学透，不是看完概念就算了，而是能在真实 Agent 里跑起来、跑稳定。  
刚好 LangChain 1.0 那会儿又没有现成的 Skills 机制，我就干脆逼自己一把：**手写一个中间件**，把 Skills 这条链路从 0 搭出来。

<u>我认为 Skills 想解决的一大问题，是工具一多，模型就开始“变笨”。</u>  
不是它不会推理，而是它的决策面被污染了——上下文里塞满工具说明，模型每轮都在几十个候选里做选择，结果就是：  
**更容易选偏、排查更慢、链路更不稳定**。我当时对实现 Skills 的直觉理解也很简单：既然问题出在“工具全量暴露”，那就模仿一下 **渐进式披露**，让模型在需要的时候才看到需要的能力。

<u>所以我第一版做得很工程，也很直接：</u>**<u>先把执行面控住</u>**。我把工具拆成两层：一层是“入口工具”，就像目录一样；另一层是“能力工具”，是真正干活的工具。  
默认我只让模型看到少量的入口工具，让大模型先回答“我想要用哪些能力”。一旦模型选中了某个入口，我就把这个选择写进会话状态，然后中间件在每次模型调用前拦一下：  
把 tools 列表动态改成“入口工具 + 当前已启用技能对应的那一小撮能力工具”。  
这版的收益其实挺快的：**<font style="color:#DF2A3F;">候选集一下子变小，模型误选明显下降，整条链路也更顺了</font>**<font style="color:#DF2A3F;">。</font>

但跑了一段时间之后，我发现这个方案有个致命问题：<u>它</u>**<u>看起来像 Skills，但容易徒有其表</u>**。  
技能会悄悄退化成“工具分组”——模型确实更不容易选错工具了，**<font style="color:#DF2A3F;">可它还是会在“怎么做才算对”上不稳定</font>**：  
同一个需求，不同对话里步骤不一致，就会导致结果不一样。

<u>这是我在实习的时候遇到过的一个实际的问题</u>，当时我的架构是 Agent + MCP，经常会出现我在写好 MCP Server 之后，还需要给 Agent 一个 Rules，教他怎么用这些工具。

同时<u>还会有一个问题，入口工具会越写越多</u>，假设是一个团队在用这个系统，那么想复用也很困难：  
比如你要改一条方法论，那么入口工具就得变，就得改代码、发版本。那一刻我才真正想明白：**<font style="color:#DF2A3F;">难点从来不是把工具藏起来，而是把经验沉淀成可复用机制的同时，提供一套运行时工具治理机制</font>**<font style="color:#DF2A3F;">。</font>

于是我没有推翻第一版，而是沿着它继续把 Skills 做“完整”。  
我把 Skills 拆成两个平面：**知识平面 + 执行平面**，但仍然用同一套 Middleware 把它们编排成闭环。

+ <u>在知识平面，我把每个技能做成一个 SkillPack</u>：  
用一份标准化说明把 workflow、适用边界、常见坑固化下来；  
中间件每轮只注入“技能目录”和最少引导，模型需要时再按需读取技能内容。  
<font style="color:#DF2A3F;">既能渐进式披露、控制 token，又能把 SOP 变成团队资产，而不是写死在代码里。</font>
+ <u>在执行平面，我保留第一版“动态装配最小工具集”的优势</u>，但把“装配依据”升级成更硬的边界：  
SkillGate 只认 `allowed_tools` 白名单。模型要执行，就必须显式启用技能；  
启用之后我才按白名单重写状态并装配最小工具集——这一步把“工具装配”从路由优化，升级成真正的执行治理。  
最终在评测里，工具误选率降低了 **80%**。

做完之后我<u>对 Skills 的理解也更收敛了</u>：  
Skills 既不是一份文档，也不是一组工具分组，它本质上是 **“可复用 SOP + 可执行边界”**。  
而 Middleware 的价值，就是把这两件事在每一轮对话里稳定落地——模型先“知道怎么做”，再“只拿到该拿的权限”。

在整个项目中，我把 enable 设计成显式授权点，所以 **权限隔离** 很好接入；  
整条链路我留了 **观测与回放** 的抓手，能把“读了什么技能、启了什么权限、调用了什么工具”串成可复盘的决策链；  
另外加了 **TTL / LRU 回收**，避免技能越开越多，工具集合又膨胀回第一天的样子。

# 动态重写
首先说一下，为什么需要动态重写，最主要的原因**工具一多以后，Agent 为什么会变笨**。  
因为<u>工具全量暴露给模型，带来的不是线性增益</u>，而是上下文膨胀、选择噪声变大、误调用变多。

我的处理方式跟最近很火的 OpenClaw 有点相似，<u>OpenClaw 不是把工具随便塞给模型</u>，  
而是把 tools 当成 first-class typed tools 来治理；而且工具是**<font style="color:#DF2A3F;">同时通过 system prompt 和 tool schema 两条通道暴露给模型的</font>**，  
所以真正的控制点，不只是 prompt 里怎么写，而是运行时到底让模型“看见什么、能调什么”。  
官方文档也明确说，如果工具不出现在系统提示或 schema 里，模型就不能调用；系统提示也是每次运行时动态组装的

所以<u>我在自己的项目里，做的也不是简单的 tool filtering</u>，而是把这件事<u>收敛成一个 </u>**<u>基于 Middleware 的能力控制面</u>**。

我把它拆成两层：一层管“模型知道什么”，一层管“模型能做什么”。

<u>先说“知道什么”</u>。

+ 在 `before_agent` 阶段，我会先<font style="color:#DF2A3F;">把 Skills 元数据加载进 state，做成一个轻量的 SkillIndex</font>，  
里面不是技能全文，而是 name、description、source、allowed_tools 这些索引信息，同时做缓存，避免每轮都去重复扫描和解析。
+ 然后到了 `wrap_model_call`，我不会把所有技能说明全塞进 prompt，而是<font style="color:#DF2A3F;">通过 PromptComposer 只注入一个技能目录</font>，  
让模型先知道“当前有哪些能力可以选、应该去哪读详细说明”。这一步本质上是在做渐进式披露，目的是把知识暴露从“全量灌输”变成“按需发现”。

<u>再说“能做什么”</u>。

如果只做到前面那一步，其实还不够，因为模型就算知道了技能目录，执行时还是可能在大工具集中误选。

所以我在 `wrap_model_call` 的另一侧，会<font style="color:#DF2A3F;">根据当前会话里已经启用的 </font>`<font style="color:#DF2A3F;">skills_loaded</font>`<font style="color:#DF2A3F;">，再结合 skill 上定义的 </font>`<font style="color:#DF2A3F;">allowed_tools</font>`<font style="color:#DF2A3F;"> 白名单，</font>  
动态计算出这一轮真正需要下发的最小工具集合，然后用 `request.override(tools=...)` 直接重写这次模型调用的 tools。  
这样模型不是一直面对全量工具，而是先选 skill，再在 skill 的边界里选 tool。

**我这里最关键的判断是：权限控制不能只靠 prompt 提醒，必须落到 tool schema 的可见性上。**   
这点其实和 OpenClaw 的 `tools.allow / tools.deny`、tool groups、provider-specific policy 是同一类思路，  
都是把“工具可见性”当成治理边界，而不是把它当成一个单纯的提示词问题。

<u>我后来复盘这块工作时，最大的体会是：</u>  
如果只做 Prompt 重写，那你得到的是“文档型 Skills”，模型知道怎么做，但执行层还是会乱；  
如果只做 Tool 重写，那又容易退化成“工具分组”，模型看到的是一组工具包，但不知道这个技能的 SOP 和边界。

**所以我最后不是在做一个技能目录，也不是在做一个工具过滤器，而是****<font style="color:#DF2A3F;">在 Middleware 里把知识披露和执行授权合成了一套统一机制</font>****。**   
这也是为什么我会把它定义成“渐进式披露 + 动态授权”的 Skills 框架。

从工程实现上讲，这套设计还有两个好处。  
第一，它是无侵入的，我不需要改业务工具本身，而是在 Middleware 这一层统一做 request rewrite；  
第二，它天然适合继续往下长，比如会话级 state、TTL/LRU 回收、防止 skill 越启越多、还有 Langfuse 的 trace/session/observation 去看 read、enable、tool call 的漏斗。  
**也就是说，我****<font style="color:#DF2A3F;">不是只想让模型这次选对工具，而是想把 Skills 变成一个可治理、可观测、可收敛的运行机制。</font>**<font style="color:#DF2A3F;"> </font>  
这个点我觉得也是我和很多只停留在 prompt 工程层面的 Agent 项目，最大的区别。



# State 缓存与回收
平时开发一想到“缓存”可能就想着上 Redis，但这个场景真正高频的其实不是业务数据查询，  
而是 **<font style="color:#DF2A3F;">Skill 元数据加载、文档读取</font>**。所以我先把它拆成了两层，分别对应不同的失效边界。

这个工作我最先关注的其实不是“怎么加一个缓存”，而是**<u>怎么把多轮对话里的技能生命周期管住</u>**。  
因为 Skills 这种机制一旦进入多轮任务，最容易失控的不是单次调用，而是状态会越跑越多：  
前面读过的技能、启用过的能力、已经装配过的工具，如果没有统一管理，最后又会回到“工具越多、模型越笨”的老问题。

所以我后来<u>把它收敛成了一套会话级状态管理，而不是零散加几个变量</u>。核心上我把状态分成三类：  
第一类是 `skills_metadata`，它代表这个会话当前“知道有哪些技能”；  
第二类是 `skills_loaded`，代表这个会话已经显式启用了哪些能力；  
第三类是 `active_tools`，代表这一轮真正要下发给模型的最小工具集。

这样我把“知道什么”“已经授权什么”“此刻能用什么”这三件事拆开了，Agent 的行为边界就清楚了。

具体运行上，<u>我是顺着 Agent 调用链做的</u>。

用户一进来，<u>先走 </u>`<u>before_agent</u>`，这里我<font style="color:#DF2A3F;">不会每轮都重新扫技能目录，而是先检查 </font>`<font style="color:#DF2A3F;">state.skills_metadata</font>`<font style="color:#DF2A3F;"> 在不在；</font>  
如果已经有了，我直接复用，避免重复做 source 遍历、目录扫描、SKILL.md 下载和 YAML 解析。  
只有新会话，或者显式刷新状态的时候，我才重新加载。这个判断很关键，**<font style="color:#DF2A3F;">因为技能库加载本质上是“相对稳定的会话上下文”，没必要每轮都做一遍。</font>**

<u>然后到了 </u>`<u>wrap_model_call</u>`，我做两件事：  
一件是知识平面，把 `skills_metadata` 格式化成一个轻量级技能目录注入到 system prompt，让模型先看到“有哪些技能、各自做什么、需要时去哪里读详情”；  
另一件是执行平面，根据 `skills_loaded` 重新计算这一轮的 `active_tools`，然后用 `request.override(tools=...)` 把本轮可见工具集直接重写掉。

这样模型每一轮看到的东西都不是固定的，而是被当前会话状态驱动出来的。

当然，`<u>active_tools</u>`<u> 不能做“只增不减”的追加式维护，而要</u>**<u>每轮基于当前状态重算</u>**。  
因为一旦用单纯 append 的思路，前面启用过但后面已经不相关的工具会一直残留，会话越长噪声越大。  
重算虽然每轮多了一点组装成本，但换来的是执行边界稳定，而且状态是可解释的，任何时候都能知道“这一轮为什么只给模型这几个工具”。

缓存这块我做了两层，不是一把梭。

<u>第一层是 </u>**<u>技能元数据缓存</u>**，缓存的是技能目录和元数据，比如名称、描述、path、allowed_tools 这些

这个我直接放在 Agent 的会话 state 里，也就是 `state.skills_metadata`。原因很简单：  
Skills Middleware 在 `before_agent()` 里本来就会先检查这个字段，如果已经存在就直接返回，不再重复扫描 sources、列目录、加载 `SKILL.md`；  
只有第一次进会话，才会走 `ls_info + load_files + parse` 这条链路。  
也就是说，<font style="color:#DF2A3F;">这一层本质上就是 </font>**<font style="color:#DF2A3F;">session 级缓存</font>**<font style="color:#DF2A3F;">，我不需要再额外引一个缓存系统</font>，直接借 Agent state 就能把最贵的重复 IO 挡掉。

<u>第二层是 </u>**<u>文档内容缓存</u>**，也就是 `SKILL.md` 的正文或者 `read_skill` 返回的关键片段。

<font style="color:#DF2A3F;">实现方式是 </font>**<font style="color:#DF2A3F;">进程内 LRU + TTL</font>**<font style="color:#DF2A3F;">，比如用 </font>`<font style="color:#DF2A3F;">cachetools.TTLCache</font>`。这里我缓存的不是“整个技能系统状态”，  
而是某个 skill 文档的解析结果，key 一般就是 `skill_path`，更稳一点可以带上 `version/hash`。

我这么做的原因是：<font style="color:#DF2A3F;">渐进式披露本身就意味着模型会按需多次读技能文档，如果每次都回 backend 去读文件，成本又会被拉回来</font>；  
而且技能文件解析本身还有文件大小限制、`yaml.safe_load`、异常跳过这些保护逻辑，重复走这条链路没有意义。

> LRU 是为了解决“会话里读过很多技能，但真正反复用的通常就那几个”，我优先保留最近最常访问的技能正文；  
TTL 是为了解决“技能内容不是永远不变的”，尤其是你支持多 source 和动态更新时，缓存不能无限期相信旧内容。
>
> 两者结合，本质上是在做一个工程上的 trade-off：<font style="color:#DF2A3F;">我既不想每次都重新 IO，也不想让模型长期拿着过期的 SOP 执行</font>。  
Skills 体系天然就有“会话内稳定、会话间刷新”的边界，我的 TTL 设计也是顺着这个边界落的。
>

**然后就是回收控制，或者说**失效策略我也分的很清楚，不然缓存很容易把技能系统做“死”。

+ **元数据缓存**的失效是显式的：要么新会话、要么清空 state、要么走 `refresh_skills`；
+ **文档缓存**我用 TTL ~~或版本号控制，技能文件变了就让 key 失效~~。

**总结一下就是我按 Middleware 的运行链路把缓存分成会话级 state 缓存元数据，进程内 LRU/TTL 缓存文档内容。  
****这样既把重复 IO 压下去了，又不会把技能更新、权限切换和工具授权做僵。**

再往深一点说，<u>这套状态管理其实也体现了我对 Skills 运行原理的理解。</u>  
我认为 Skills 不是“给模型多塞一段说明书”，而是一个分两步走的机制：先让模型在知识层知道“有什么能力”，再让它在执行层只拿到“当前该有的权限”。  
前者靠 `skills_metadata + read_skill cache`，后者靠 `skills_loaded + active_tools`。  
如果没有会话级状态，这两层是断开的；模型这一轮读过什么、下一轮该不该继续给它权限，系统其实说不清楚。

所以这块工作表面上看是在做 state、LRU、TTL，实际上我是在给 Skills 框架补一套**多轮运行时控制面**。  
这样做完之后，多轮对话里不会每轮都重新扫技能库，也不会因为启用过的能力不回收而再次膨胀；  
同时 Agent 的调用链是连贯的：`before_agent` 负责加载稳定上下文，`wrap_model_call` 负责按当前状态重写 prompt 和 tools，  
`read_skill` 走正文缓存，`enable_skill` 更新授权状态，下一轮再基于新状态重算最小工具集。





# Langfuse 监控
监控系统链路的第一步，是先把 Agent 这条链路抽象清楚。因为如果链路没抽象清楚，后面埋点一定是散的，最后只能看到一堆日志，看不到行为。  
我的做法是<u>先把整个执行过程收敛成五个关键阶段</u>：发现技能、读取 SOP、显式授权、调用工具、完成任务。然后我把这五段一个一个映射到 Langfuse 里：

一次用户请求对应一个 trace，多轮对话用 session 串起来，`skill.read`、`skill.enable`、`tool.call`这些关键动作都落成 observation。  
Langfuse 本身就是按 trace、session、observation 这个模型来组织 LLM 应用链路的，而且 metadata、tags 这些属性可以直接挂在链路上，后面过滤和审计会很方便。

具体落地上，我是放在 Skills 架构的几个关键节点做的，而不是 everywhere 都打上日志。

<u>第一层是在 </u>`<u>before_agent</u>`<u> 和 </u>`<u>wrap_model_call</u>`<u> 这两个中间件钩子上</u>：  
前者负责初始化 request_id、session_id、user_id、tenant_id 这些上下文，后者负责记录本轮注入了哪些 skill 元数据、当前可见工具集被怎么改写了。  
因为我这个 Skills 架构本质上是“知识平面 + 执行平面”两层，知识平面是技能目录和 `SKILL.md` 的渐进式披露，执行平面是 `enable_skill` 之后的工具白名单装配，  
所以我<font style="color:#DF2A3F;">必须同时观测“模型知道了什么”和“模型实际能做什么”，不然你根本分不清问题到底出在 skill 描述、授权策略，还是工具绑定上。</font>

<u>第二层是 </u>`<u>skill.read</u>`<u> 和 </u>`<u>skill.enable</u>`。这两个点我认为是最有价值的，因为它们把“文档型 Skills”和“执行型 Skills”真正连起来了。  
`skill.read` 我会记 skill 名、是否命中缓存、读了多长内容；  
`skill.enable` 我会记启用了哪个 skill、启用前后 `skills_loaded` 怎么变、放开了哪些 tools、最终决策是 granted 还是 denied、拒绝原因是什么。  
这样我后面就能了解两个很关键的问题：  
<font style="color:#DF2A3F;">第一，模型有没有真的理解这个 skill；第二，系统有没有在正确的时机给它正确的能力。</font>

<u>第三层是 </u>`<u>tool.call</u>`。这里我不是只记成功失败，而是专门把“误调用、重试、失败归因”拆开做了。  
误调用我会分成三类：

+ 一种是 tool 根本不在当前白名单里，这说明授权边界有问题；
+ 一种是 tool 在系统里合法，但不属于当前 skill 域，这更像路由错了；
+ 还有一种是 tool 选对了，但参数 schema 不对，这往往是模型抽参不稳定。

重试我也不是简单记次数，而是把<font style="color:#DF2A3F;">第几次重试、第一次为什么失败、最后有没有恢复都挂到 observation 上。</font>  
这样后面看一条 trace 的时候，不是只看到“失败了”，而是能看到它到底是选错工具、参数错、超时，还是白名单配宽了。

<u>再往上就是统一指标</u>。我没有把它做成一堆零散指标，而是围绕一个漏斗去看：`shown -> read -> enable -> tool -> task`。  
也就是技能被展示了多少次、被读取了多少次、读完后有多少真的被启用、启用后工具调用有没有成功、最后任务有没有完成。这样一来，

+ 如果某个 skill `shown` 很高但 `read` 很低，我会判断是技能目录描述有问题，导致没有被选中；
+ 如果 `read` 很高但 `enable` 很低，通常是 `SKILL.md` 写得太重或者边界不清；
+ 如果 `enable` 很高但任务完成率低，那我会优先怀疑白名单收敛得不够好，或者 skill 拆得太粗。

Langfuse 这边我会直接用 generation 记录 token、cost、latency，用 score 去挂任务完成、路由正确、是否遵守最小工具集这些结果，再通过它的查询和聚合接口做后续分析。

这块工作我觉得真正的价值不是“我接了 Langfuse”，而是我把 Agent 的运行过程从黑盒拆成了一个可解释、可回放、可优化的链路。  
后面我去做 Skills 的拆分/合并、`allowed_tools` 的白名单收敛，或者排查某次多轮任务为什么跑偏，都是直接基于这套链路做判断，而不是靠猜。  
**这也体现了我对 Agent 的理解不是停留在 prompt 和 tool 调用层，而是把它当成一个有状态、有授权边界、能被观测和治理的执行系统来做。**

****

## Langfuse 具体怎么用
我理解 Langfuse 不只是一个“看日志的面板”，还可以把它当成 Agent 调用链的观测入口来看。  
Langfuse 的核心就是 trace、span、generation，trace 是一次完整请求流程，span 是其中的普通步骤，generation 是一次 LLM 调用。

具体落地时，我会根据场景选不同接法。

+ 最轻量的方式是直接把 OpenAI <font style="color:#DF2A3F;">SDK 替换成 </font>`<font style="color:#DF2A3F;">from langfuse.openai import openai</font>`，  
这样 prompt、completion、latency、token 和 API error 都会自动被追踪下来，这适合我先快速把线上链路接起来。
+ 如果是我自己写的函数式链路，比如 RAG 或 Agent 的某几个核心步骤，我会<font style="color:#DF2A3F;">用 </font>`<font style="color:#DF2A3F;">@observe()</font>`<font style="color:#DF2A3F;">包住关键函数，把普通步骤记成 span，把模型调用记成 </font>`<font style="color:#DF2A3F;">as_type="generation"</font>`，  
这样链路层级会更清楚，然后短生命周期任务最后要 `get_client().flush()`，不然数据可能还没刷出去。  
还能通过 `capture_input=False`、`capture_output=False` 控制大数据场景下的观测成本
+ 如果项目本身已经是 LangChain 或 LangGraph 了，我一般会<font style="color:#DF2A3F;">优先用 </font>`<font style="color:#DF2A3F;">CallbackHandler</font>`，  
在 `chain.invoke()` 或 `app.invoke()` 的 `config` 里挂 callbacks，  
同时把 `langfuse_user_id`、`langfuse_session_id`、`langfuse_tags` 一起带进去。  
这样我既能追踪链路，又能按用户、会话、标签去查问题，这对多轮 Agent 排障特别有用。

除了 tracing，<font style="color:#DF2A3F;">Langfuse 还能做 Prompt 管理，可以把 prompt 放到 Langfuse 里做版本和标签管理</font>，  
比如 production、staging、latest，这样提示词变更不会散在代码里。  
Langfuse 还能做评估，可以用 `langfuse.score()` 给 trace 打分，支持 LLM-as-Judge、人工标注、用户反馈、自定义评分。

## 本地部署 Langfuse 的流程是什么？
先跑起来其实很直接：先安装 SDK，然后配三类环境变量——`LANGFUSE_SECRET_KEY`、`LANGFUSE_PUBLIC_KEY`、`LANGFUSE_HOST`。  
如果是本地实例，`LANGFUSE_HOST` 就指到本地地址。

真正部署服务端，下载官方 docker-compose.yml。最核心的不是只起一个 langfuse 容器，而是把它依赖的数据层一起起起来：

+ Langfuse 主服务
    - `langfuse-web`：Web 界面和 API
    - `langfuse-worker`：后台任务
+ Postgres，做事务型数据存储
+ ClickHouse，做分析型存储
+ Redis，做缓存

然后在 langfuse 服务里配 `DATABASE_URL`、`NEXTAUTH_SECRET`、`SALT`、`NEXTAUTH_URL` 这些环境变量，最后直接 `docker-compose up -d` 启动。

Langfuse 不只是一个单机调试工具，往生产走时它是一个有在线服务层、异步 worker 和多种存储分工的数据系统。  
本地部署的最小流程就是“准备 Compose → 起 Langfuse + Postgres + ClickHouse + Redis → 配 host 和 key → 启动服务 → 应用侧把 `LANGFUSE_HOST` 指到本地实例”。

# 回放评测验证
<u>第一步，我先做一套</u>**<u>离线回放数据集</u>**，本质上就是 50 多条“工具选择题”。  
每个样本都是一个结构化对象 ，我都会标清楚三件事：用户问题是什么，期望应该选哪个工具，为什么是这个工具。  
这里我不会只放简单题，我会刻意混一些“相似工具干扰题”，比如同样都和检索、读取、处理相关，但边界不完全一样，这样才能真的测出误选。****

<u>第二步，我会做</u>**<u>三组对照</u>**<u>。</u>  
第一组是全量 tools 直接暴露，这是最原始的 baseline；第二组是只有技能目录和 SKILL.md，也就是模型知道“怎么做”，但工具还是很多；  
第三组才是我的最终方案：先通过技能目录发现能力，再按需读 SKILL.md，最后通过 `enable_skill` 和白名单把本轮工具集收缩到最小。  
这样做的好处是，我能把效果拆开看清楚：到底是“知识”带来了提升，还是“缩小工具集”带来了提升，还是两者叠加才真正有效。

<u>第三步是</u>**<u>固定变量跑回放</u>**。  
我会把模型版本、prompt 模板、temperature、工具 mock、超时和重试策略都固定住，唯一改变的就是工具暴露方式和技能机制。  
每条样本我都会记录几个指标：首个工具是不是选对了，最后任务有没有完成，中间有没有换工具、重试，以及 token、latency、可见工具数。

<u>至于怎么证明误选率真的下来了</u>，我当时测试的时候是让 大模型造了几十个 Tools 并且混入当前的工具里面，然后测试了 50 次，完全暴露的选错了将近 30 次  
而采用这一套渐进式披露 SOP + 显式工具授权的框架之后，只错了五六次。  
但是我想说的是，这不会是一个绝对数字，因为错误类型有很多，所以波动会比较大，但至少能说明趋势是非常明显的。  

在这个基础上，我还会<u>补一层</u>**<u>混淆矩阵</u>**<u>，这是我觉得很关键的一步</u>。因为只看“错了多少次”不够，你还得知道“到底错成了谁”。

我的做法是把行设成 `expected_tool`，列设成 `predicted_tool`，每跑完一批回放，就统计一次首个工具调用的分布。  
这样对角线表示选对了，非对角线就表示高频混淆对。比如如果本来该选 `search_docs`，却经常被选成 `search_web`，  
那我就知道这不是随机错误，而是这两个工具在模型眼里语义边界没拉开。  
<u>混淆矩阵对我有两个作用</u>：

第一，它能告诉我高频误选是不是集中在少数几对工具之间；  
第二，它能指导我后面的优化动作——到底是该改 tool description、拆 skill，还是缩小白名单。  
**也就是说，误选率告诉我“有没有变好”，混淆矩阵告诉我“为什么还没完全变好”。**

<u>然后第四步，我会</u>**<u>做误差分析，而不是只报一个百分比</u>**

因为误选不只有一种。有的是真选错工具了，有的是工具选对了但参数传错了，报错后模型又换了个更宽松但不对的工具；  
还有一类是它本来应该先读技能文档或者先澄清问题，但它自信过头，直接开跑。  
<font style="color:#DF2A3F;">所以我会把错误分桶，至少拆成语义混淆、参数错误、没遵循 workflow、描述误导、执行反馈处理失败这几类。</font>

<u>这时候混淆矩阵和错误分桶是配合起来看的</u>：  
混淆矩阵负责找出“哪几个工具总被搞混”，错误分桶负责解释“它为什么会搞混”。  
这样我才能知道，误选率下降到底是因为候选集少了，还是因为 SKILL.md 里的 workflow 真起作用了，还是因为某几个工具描述本来就写得有问题。  

还有一点我想说的是，“缩小工具集 + 给了 workflow，就不会误选了”是不可能的。**<u>误选永远不会彻底消失</u>**<u>。</u>原因很简单，

第一，缩小候选集只能降噪，不能消除语义歧义；如果剩下的 10 个工具里还有 3 个特别像，模型还是可能选错。  
第二，SKILL.md 提供的是 workflow，它更像“建议”和“经验”，不是硬约束；模型可能没读、读了没照做，或者上下文长了以后注意力漂移。  
第三，很多误选表面看是选错工具，本质其实是参数没传对、执行报错以后模型错误修复了。

所以我最后不是只做了知识平面，而是把执行平面也补上了：用显式授权和最小工具集把能力边界收紧，再配合回放评测和 Langfuse 观测去持续修正。  
这也是我觉得这个项目真正有工程价值的地方，**我不是在赌模型更聪明，而是在用机制让它更不容易犯错。**

## 怎么根据错误分桶修正
根据错误类型去修正系统也很重要，**我最关键的判断就是，****<font style="color:#DF2A3F;">先分清楚问题发生在认知层、流程层，还是执行边界层。</font>**

<u>第一类我会看是不是</u>**<u>语义混淆型错误</u>**。

比如两个工具都带 search、read、parse 这种词，模型经常把 A 选成 B。这种我**<font style="color:#DF2A3F;">优先改 tool description，而不是先动 skill</font>**。改 description 我有几个原则。

+ 第一，不写大而全的介绍，而是写“这个工具最适合解决什么问题、不适合解决什么问题”。
+ 第二，我会刻意加**边界和反例**，也就是告诉模型“遇到 XX 不要用我”
+ 第三，我会减少几个工具之间重复出现的关键词，避免它只按词面匹配。

如果一个工具描述改完之后，混淆矩阵里它和另一个工具的互相误选明显下降，那我就知道这个问题主要在语义层。

<u>第二类我会看是不是流程缺失型错误</u>。这种错误不是模型不知道工具名，而是它没有沿着正确的 SOP 往下走，不清楚这类任务应该先做什么、后做什么。

比如明明应该先阅读材料、整理要点、确认信息是否完整，再进入输出阶段，它却在材料还没吃透的时候就直接开始生成正式内容。  
这个时候我不会直接去改 description，因为 description 只能回答“这个工具是干什么的”，回答不了“这类任务应该按什么顺序做”。

<font style="color:#DF2A3F;">我会先判断是不是一个 skill 里混了不同性质的 SOP</font>。  
如果一个 skill 同时承载了“阅读材料并整理要点”和“根据材料生成正式输出”两段流程，我通常会拆开，  
因为前者的核心是理解、提取和归纳，后者的核心是组织、表达和满足格式约束，它们的目标和判断重点并不一样，  
尤其是存在多轮运行的时候，放在一起模型很容易还没完成理解就提前进入生成。

但如果问题是同一条 SOP 被切得太碎，模型在多个 skill 之间来回跳，反而拿不到完整上下文，那我会选择合并，而不是继续拆。

所以我后来对 skill 拆分的理解是：不是默认越细越好，而是让每个 skill 只承载一条边界清晰、节奏稳定的决策链路。

<u>第三类我会看是不是</u>**<u>边界过宽型错误</u>**。

也就是说，模型其实不一定完全理解错了，<font style="color:#DF2A3F;">但因为这个 skill 挂的工具太多，它总能“顺手”摸到一些相邻能力，最后走偏</font>。  
这种我就不再纠结模型为什么想这么做，而是直接收白名单。收白名单我一般会做三步。

+ 先看这个 skill 在真实回放里，哪些工具几乎从来没被正确使用，或者一出现就容易把任务带歪；这种工具我会优先移出 allowed_tools。
+ 然后我会区分“核心工具”和“辅助工具”，核心工具保留，辅助工具只有在特别明确的 skill 里才挂进去。
+ 最后我会看风险等级，高风险工具，比如涉及到写库、删数的工具，尽量不出现在宽泛 skill 里，而是放到更窄、更需要显式授权的 skill 下面。

这样做的本质不是优化体验，而是<font style="color:#DF2A3F;">把“模型可行动的空间”压缩到足够小。因为 workflow 再好，也只是软约束；真正能兜底的还是执行边界。</font>

<u>还有一类我特别注意的是</u>**<u>伪误选</u>**。表面上看是选错工具，实际上是第一个工具选对了，但<font style="color:#DF2A3F;">参数没传对，报错以后模型又换了个错误但更宽松的工具。</font>

这种如果不分桶，就很容易误以为 description 有问题，结果越改越乱。  
我的处理方式一般是把它归到“参数错误/反馈修复失败”，优先补 skill 里的检查清单，比如调用前必须确认哪些字段、报错后先修参数还是换工具。  
只有<font style="color:#DF2A3F;">当它真的是一上来就选错，我才去动 description 或白名单。</font>

<u>所以整体上，我的迭代路径其实很明确</u>：

+ 如果是**看不懂工具**，我改 description；
+ 如果是**不知道怎么做**，我拆 skill、补 workflow；
+ 如果是**知道大概怎么做但手伸太长**，我收白名单。

这样改的好处是，每次迭代都能对应回某一类错误，而不是把所有问题都糊成一句“继续优化 prompt”。  
**这也是我比较强调的一点：Agent 的稳定性不是靠某一版 prompt 突然变神，而是靠错误分桶之后，分别在语义、流程、边界这三层持续收敛。**

~~所以我把风控收敛成一个核心原则：~~~~**工具可见性就是权限边界**~~~~。落地上分三层做。~~

~~~~**授权点本质是一个可审计的闸门**~~~~——它会先做策略校验（比如这个 skill 是否允许当前用户/环境），然后读取 skill 的 ~~`~~allowed_tools~~`~~ 白名单，只把这份白名单对应的工具装配进可见工具集。  
~~~~这样模型即使学会了调用姿势，也拿不到白名单之外的能力，风控边界是硬的，不靠模型自觉。~~

**<font style="color:#DF2A3F;">第三层是“输入治理 + 可观测闭环”。因为 skills 不是我一个人写完就不变了，它来自 base/team/project/user 多来源，还会发生覆盖，所以我把技能库当成不可信输入：</font>**

+ **<font style="color:#DF2A3F;">加载阶段要做安全解析和治理——文件大小限制、防止恶意超大文件拖垮系统；</font>**
+ **<font style="color:#DF2A3F;">YAML 用 </font>**`**<font style="color:#DF2A3F;">safe_load</font>**`**<font style="color:#DF2A3F;">，缺字段就跳过，只允许解析到基础类型（dict/list/str/number）；</font>**
+ **<font style="color:#DF2A3F;">描述截断、清洗、分隔符封装（把它当 data，而不是 instruction），避免Prompt Injection；</font>**
+ **<font style="color:#DF2A3F;">解析失败只告警不影响其他技能。</font>**

同时我会把每次 enable/read/工具调用都打点：哪些技能经常启用但失败率高、哪些工具误调用多、一次任务里工具集膨胀到什么程度——这些数据我会接到 Langfuse 做漏斗和失败归因，反过来迭代白名单和 SOP。

最后我还加了一个很工程化的小收口：技能不是越开越多，启用后如果 N 轮没用就 TTL/LRU 回收，重新计算可见工具集，把“最小权限”贯彻到底。





## Claude Skills 是去年 10 月中就推出的，为什么现在才突然火起来？  
10 月份的时候它只是“给程序员的工具”，而 1 月份它变成了“谁都能用来干活的自动化”，面向的用户更广了，出圈了自然就火了。  
我认为主要原因是：  
第一，Skills 能配合 bash/脚本直接操作电脑，不需要编程知识也能自动化完成任务，所以用户面一下子变大；  
第二，多个 Skills 能互相调用，用自然语言就能把流程串起来，不需要会写工作流；  
第三，Skill 打包分发很轻，加上 12/18 规范开放、生态跟进，大家更容易复用和传播。

1 月正好叠加“产品更好用 + 非编程场景跑通 + 传播更容易”，主要是降低了门槛，所以突然爆了。
