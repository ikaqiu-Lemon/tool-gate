# Claude Code 工具治理插件 -- 开发计划

> 版本：v2.2 | 更新日期：2026-04-15
> 基于 Skill-Hub 项目经验、Claude Code 官方插件规范（2026）、ivan-magda/claude-code-plugin-template 模板标准修订

---

## 一、项目概述

### 1.1 项目名称

`tool-governance-plugin`（暂定）

### 1.2 项目定位

一个 **Claude Code 插件形态的 Python 工具治理运行时**，而非简单的权限面板或工具过滤器。

核心定位：**人定边界，系统执法，风险升级** -- 用户定义治理策略，系统在运行时自动执行能力暴露、授权校验、生命周期回收，高风险动作显式升级到人工确认。

### 1.3 核心目标

1. **按任务动态缩减工具面**：在不同任务阶段只下发当前需要的最小工具集合
2. **显式授权与知识披露**：通过 `list_skills` / `read_skill` / `enable_skill` 流程，将"理解能力"与"执行权限"分离（双平面架构）
3. **每轮状态驱动的 Prompt/Tool 重写**：每轮用户消息后，基于当前会话状态重新注入技能目录摘要和更新后的 active_tools 提示（对应 Skill-Hub 中的 `wrap_model_call`）
4. **会话状态管理与回收**：维护技能元数据、已启用技能、当前可见工具等会话级状态，通过 TTL/LRU 回收
5. **Stage 粒度支持**：同一 Skill 可按阶段暴露不同工具集，支持 `change_stage` 阶段切换
6. **可观测与审计闭环**：关键节点（skill.read/enable、tool.call）结构化记录，支持漏斗分析和误调用分桶
7. **符合 Claude Code 插件规范**：严格遵守官方插件标准，可通过插件市场分发

### 1.4 技术栈

| 类别 | 选型 |
|------|------|
| 语言 | Python 3.11+ |
| 核心框架 | LangChain |
| 数据模型/配置 | Pydantic v2 |
| 缓存 | cachetools (LRU/TTL) |
| MCP Server | mcp Python SDK (stdio) |
| HTTP Hook 服务 | FastAPI (可选) |
| 持久化 | SQLite（`${CLAUDE_PLUGIN_DATA}`） |
| 测试 | pytest + pytest-asyncio |
| 包管理 | pyproject.toml + uv/pip |
| 观测审计（可选） | Langfuse |

### 1.5 关键设计决策

> **[不确定项]** 以下决策基于当前（2026-04）对 Claude Code 插件系统的理解，部分行为可能随版本更新变化。

1. **固定元工具面 + 服务端子能力分发**：由于 Claude Code 对 MCP `tools/list_changed` 动态刷新的支持尚不完全稳定，本项目采用"固定少量元工具 + `run_skill_action` 服务端分发"的策略，而非依赖会话内动态增减 MCP 工具列表。
2. **Hook 类型选择**：优先使用 `command` 类型 hook 调用 Python 脚本，通过 stdin/stdout JSON 通信；如需更复杂的状态管理，可选用 HTTP hook + FastAPI 本地服务。
3. **插件级 hooks.json**：hooks 配置放在插件根目录 `hooks/hooks.json`，随插件分发，与用户现有 hooks 并行执行。

### 结构说明

- `.claude-plugin/plugin.json` 是唯一放在 `.claude-plugin/` 目录内的文件
- `skills/`、`hooks/`、`agents/`、`.mcp.json` 必须放在插件根目录，不在 `.claude-plugin/` 内
- Python 源码放在 `src/tool_governance/`，遵循 Python 标准项目布局（src layout）
- `${CLAUDE_PLUGIN_ROOT}` 用于 hook 命令中的路径引用，确保跨平台可移植性

## 五、风险与不确定项

> v2.1 新增风险项已用 **[NEW]** 标记。

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| Claude Code 插件 API 变动 | Hook 事件格式或 plugin.json schema 可能随版本更新变化 | 持续关注官方文档，将适配逻辑集中在 `hook_handler.py` 和 `mcp_server.py` |
| MCP 工具动态刷新不稳定 | 无法在会话中动态增减 MCP 工具 | 采用固定元工具面 + `run_skill_action` 服务端分发策略 |
| Python 环境依赖 | 用户机器可能未安装 Python 或版本不兼容 | 在 README 中明确 Python 版本要求；未来可考虑 Docker 方案 |
| Hook 处理延迟 | 过高的 hook 延迟会影响 Claude Code 使用体验 | 控制单次 hook < 50ms；使用进程内缓存减少 IO |
| `${CLAUDE_PLUGIN_DATA}` 路径行为 | 不同平台/版本下的具体路径可能不同 | 测试覆盖 Windows/macOS/Linux 三端 |
| 插件 hooks 与用户 hooks 的交互 | 插件 hooks 与用户已有 hooks 并行执行时可能产生冲突 | 治理插件的 hook 仅做检查和状态更新，不修改用户行为 |
| **[NEW]** UserPromptSubmit hook 延迟累积 | 每轮消息都触发 hook，频率高于 SessionStart | prompt_composer 和 tool_rewriter 使用缓存结果，避免重复计算 |
| **[NEW]** 无法直接覆盖工具列表 | Claude Code 不支持 request.override(tools=...)，只能"软引导+硬拦截" | UserPromptSubmit 注入 additionalContext 引导 + PreToolUse deny 硬拦截；效果需实测验证 |
| **[NEW]** Stage 机制的 SKILL.md 格式 | stages 字段不是 Claude Code 官方 SKILL.md 标准的一部分 | 作为自定义 frontmatter 字段，不影响 Claude Code 原生解析 |

## 五-A、Skill-Hub 概念到 Claude Code 插件的映射

> 本插件将 Skill-Hub 项目的治理机制迁移到 Claude Code 插件体系。以下是关键概念的映射关系：

| Skill-Hub 概念 | Claude Code 插件等价物 | 说明 |
|----------------|----------------------|------|
| `before_agent` | **SessionStart hook** | 会话开始时恢复状态、构建索引、回收过期 grant |
| `wrap_model_call`（每轮） | **UserPromptSubmit hook** | 每轮用户消息后注入技能摘要和 active_tools 提示 |
| `request.override(tools=...)` | **PreToolUse hook (deny)** + **additionalContext 引导** | Claude Code 不支持直接覆盖工具列表，改为 hook 拦截 + prompt 引导的组合策略 |
| `PromptComposer` | **prompt_composer.py** → UserPromptSubmit 的 additionalContext | 将技能目录和工具提示注入模型上下文 |
| LangChain state | **SessionState + SQLite** | 会话级状态持久化到本地数据库 |
| Redis（跨实例续跑） | **SQLite**（单机足够） | Claude Code 插件是本地运行，不需要跨实例；SQLite 可满足持久化需求 |
| Langfuse 观测 | **Langfuse 集成** + **SQLite 审计日志** | 双通道：Langfuse 负责链路追踪，SQLite 负责结构化审计 |
| `read_skill` / `enable_skill` | **MCP 元工具** | 通过 MCP Server 暴露给模型调用 |
| `allowed_tools` 白名单 | **PreToolUse hook 硬拦截** | 不在白名单的工具调用被 hook deny |
| `change_stage` | **MCP 元工具 change_stage** | 模型显式调用切换阶段 |

### 关键差异

1. **无法直接覆盖工具列表**：Skill-Hub 的 `request.override(tools=...)` 可以直接重写模型看到的工具集。Claude Code 插件无法做到这一点。本项目的替代方案是：
   - 通过 **UserPromptSubmit hook** 的 additionalContext 告诉模型"当前可用的工具是..."
   - 通过 **PreToolUse hook** 硬拦截不在白名单的工具调用
   - 两者组合实现"软引导 + 硬拦截"的等效效果

2. **MCP 工具列表固定**：由于动态刷新不可靠，本项目使用固定的 8 个元工具 + `run_skill_action` 服务端分发，而非动态增减 MCP 工具。
