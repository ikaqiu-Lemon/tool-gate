# Claude Code 工具治理插件 -- 需求文档

> 版本：v1.2 | 更新日期：2026-04-16
> 配套文档：[开发计划](./开发计划.md) | [技术方案文档](./技术方案文档.md)

---

## 1 项目背景

### 1.1 问题陈述

Claude Code 支持通过插件（Plugins）、MCP 服务器（MCP Servers）和技能（Skills）扩展能力。随着扩展生态快速增长，代理模型在每轮推理中面对的候选能力集急剧增加，产生以下工程问题：

1. **上下文膨胀**：大量工具描述占据上下文窗口，降低模型对任务本身的关注度
2. **工具误选**：候选工具越多，语义相近的工具越容易被模型混淆，误选率上升
3. **决策不稳定**：同一任务在不同会话中可能因工具组合不同而产生不一致的执行路径
4. **权限边界模糊**：所有已安装工具一次性暴露，缺乏按任务、按会话、按风险等级动态控制的手段
5. **状态缺失**：当前 Claude Code 没有内置的"能力生命周期"管理，工具一旦安装即永久可见

### 1.2 项目目标

开发一个 Claude Code 插件，在模型与工具之间增加运行时治理层，实现：

- **渐进式披露**：模型先看到技能目录，按需读取 SOP，再显式授权
- **动态授权**：基于任务类型、风险等级、用户策略决定是否启用某项能力
- **状态回收**：授权具有生命周期（TTL/turn/session），过期自动回收

### 1.3 项目定位

**人控优先的自动化治理运行时**：用户定义治理边界（策略），系统在运行时自动执行能力暴露、授权校验、生命周期推进，高风险动作显式升级到人工确认。

- 策略定义阶段：偏手动（用户设定规则）
- 运行时治理阶段：偏自动（系统执行规则）
- 升级审批阶段：偏人工（高风险需确认）

## 2 用户角色

| 角色 | 描述 | 核心诉求 |
|------|------|----------|
| **插件使用者** | 在 Claude Code 中执行编码、调试、知识任务的开发者 | 工具使用准确、安全、高效；不希望被无关工具干扰 |
| **团队管理员** | 配置团队级策略、审批高风险操作的管理角色 | 统一治理规则；可审计；合规 |
| **插件开发者** | 维护治理插件本身，或基于此框架二次开发的开发者 | 代码可维护；架构解耦；易于扩展 |

## 3 治理对象

本插件需要管理的治理对象包括以下 6 类，覆盖 Claude Code 当前扩展生态中的主要能力类型：

| 治理对象 | 来源 | 说明 |
|----------|------|------|
| **内置工具（Built-in Tools）** | Claude Code 自带 | Bash、Read、Edit、Write、WebFetch、WebSearch、Agent 等 |
| **MCP 工具（MCP Tools）** | 外部 MCP Server | 通过 MCP 协议接入的第三方工具 |
| **技能（Skills）** | 插件或项目级 skills/ 目录 | 扩展 Claude 能力的结构化 SOP + 工具白名单 |
| **子代理（Subagents）** | Agent 工具创建 | 带独立上下文和工具限制的子任务执行器 |
| **插件组件（Plugin Components）** | 已安装的插件 | 一个插件可能包含 skills + hooks + MCP + agents |
| **MCP Prompts/Resources** | MCP Server | 影响模型"知道什么"的上下文数据和消息模板 |

> **[不确定项]** 当前版本优先治理 Skills 和 MCP Tools 这两类对象。内置工具的治理（如 Bash 命令白名单）可通过策略配置覆盖但不做深度改造；子代理的工具限制利用 Claude Code 原生的 `tools`/`disallowedTools` 机制。

## 4 治理维度

完整的工具治理需要覆盖以下 8 个维度。V1 版本聚焦前 6 个维度，第 7、8 维度为可选/后续增强。

### 4.1 可见性治理

**问题**：Claude 现在看到了哪些能力？是否过多？

**需求**：
- 会话开始时，Claude 仅看到少量元工具（list_skills、read_skill、enable_skill 等）
- 技能目录以摘要形式注入 system prompt，而非全文
- 未启用的技能对应的底层工具不出现在模型可见范围

### 4.2 语义治理

**问题**：Claude 如何理解这些能力？是否会混淆？

**需求**：
- 每个技能有清晰的用途定义、适用场景、边界说明
- 技能之间的职责不重叠，或重叠部分有明确优先级说明
- 通过 `read_skill` 返回的 SOP 帮助模型理解该类任务的正确流程

### 4.3 资格治理

**问题**：在当前任务中，哪些能力有资格被使用？

**需求**：
- 支持按任务类型推荐候选技能集合
- 不相关的技能即使已安装也不进入当前任务的候选池
- 子代理的能力面可以独立于主会话进行裁剪

### 4.4 授权治理

**问题**：谁批准了某项能力的使用？

**需求**：
- 低风险技能可自动授权（auto grant）
- 中风险技能需要模型显式调用 `enable_skill` 并记录原因
- 高风险技能需要用户确认（通过 Claude Code 的权限提示）
- 每次授权记录：授权者、对象、范围、时间、过期条件

### 4.5 生命周期治理

**问题**：能力什么时候生效，什么时候回收？

**需求**：
- 授权具有作用域：`turn`（单轮）或 `session`（会话级）
- 授权具有 TTL：到期自动失效
- 会话结束时清理所有临时授权
- 支持显式撤销（`disable_skill`）

### 4.6 组合治理

**问题**：多个能力拼在一起是否会越权？

**需求**：
- 每个技能通过 `allowed_tools` 白名单限定可使用的工具范围
- 只有 `skills_loaded` 中技能对应的工具才会出现在 `active_tools` 中
- PreToolUse hook 作为硬拦截，阻断白名单外的工具调用

### 4.7 信任治理（V2）

**问题**：能力的来源是否可信？

> 后续版本考虑：区分官方工具、团队内部插件、第三方 MCP Server 的信任等级，并据此设置默认风险等级。

### 4.8 审计与观测治理

**问题**：事后能否讲清楚发生了什么？

**需求**：
- 记录关键操作：skill.list、skill.read、skill.enable、skill.disable、tool.call、grant.expire、grant.revoke、stage.change
- 每条记录包含：时间戳、session_id、skill_id、tool_name、决策结果、原因
- **漏斗指标**：记录 shown → read → enable → tool → task 各阶段数据，定位链路瓶颈
- **误调用分桶**：区分三类误调用：白名单外调用、域内选错工具、工具正确但参数错误
- 存储在 SQLite 中，可查询和导出
- 接入 Langfuse 实现全链路 trace/session/observation 串联

## 5 功能需求

### 5.1 核心功能（V1 必须实现）

#### F1：技能目录获取 -- `list_skills`

- 返回所有可用技能的摘要列表
- 每个条目包含：skill_id、name、description、risk_level、is_enabled
- 数据来源：扫描 skills/ 目录 + 缓存

#### F2：技能阅读 -- `read_skill`

- 输入：skill_id
- 返回：技能的完整 SOP（适用场景、步骤说明、风险提示、allowed_tools 列表）
- 读取结果缓存（LRU/TTL），避免重复 IO

#### F3：技能启用 -- `enable_skill`

- 输入：skill_id, reason（可选）, scope（turn/session）, ttl（秒）
- 行为：
  1. 策略引擎评估是否允许启用（风险等级、策略配置）
  2. 允许则创建 Grant 记录，更新 skills_loaded
  3. 重算 active_tools
- 输出：授权结果（granted/denied）+ 允许的工具列表

#### F4：技能禁用 -- `disable_skill`

- 输入：skill_id
- 行为：撤销对应 Grant，从 skills_loaded 移除，重算 active_tools

#### F5：授权状态查询 -- `grant_status`

- 返回当前会话的所有活跃 Grant 列表
- 每个条目包含：grant_id、skill_id、scope、ttl、expires_at、status

#### F6：技能动作执行 -- `run_skill_action`

- 输入：skill_id, op（操作名）, args（操作参数）
- 行为：
  1. 验证 skill 是否已启用且 grant 未过期
  2. 验证 op 是否在该 skill 的 allowed_ops 中
  3. 分发到实际执行逻辑
- 输出：执行结果

> **[不确定项]** `run_skill_action` 的实际执行逻辑取决于具体的技能定义。V1 版本提供分发框架和几个示例 skill（如 repo-read、code-edit），实际工具调用可能需要委托给 Claude Code 的内置工具或其他 MCP Server。具体的委托机制需在实现阶段验证。

#### F7：每轮 Prompt/Tool 重写（UserPromptSubmit Hook）

- 每轮用户消息发送后、模型响应前触发
- 调用 PromptComposer 生成 additionalContext：技能目录摘要 + 当前活跃工具 + 授权引导
- 调用 ToolRewriter 基于最新 skills_loaded + current_stage 重算 active_tools
- 回收本轮已过期的 grant
- **此 hook 是"每轮状态驱动重写"的核心，对应 Skill-Hub 的 wrap_model_call**

#### F8：会话初始化（SessionStart Hook）

- 恢复或初始化会话状态
- 回收过期 Grant
- 构建技能索引（首次）或从缓存恢复
- 通过 additionalContext 注入技能目录摘要和使用提示

#### F9：工具调用拦截（PreToolUse Hook）

- 检查被调用的工具是否在当前 active_tools 中
- 不在则返回 deny 或 ask
- 在则返回 allow

#### F10：调用后处理（PostToolUse Hook）

- 记录工具调用的审计日志（含结构化字段：event_type、decision、error_bucket）
- 更新 skill_last_used_at 时间戳

#### F11：阶段切换 -- `change_stage`

- 输入：skill_id, stage_id
- 行为：更新 LoadedSkillInfo.current_stage，重算 active_tools
- 前提：skill 必须已启用且定义了 stages
- 用途：同一 Skill 按任务阶段暴露不同工具集（如"诊断阶段"只给读工具，"执行阶段"给写工具）

#### F12：技能索引刷新 -- `refresh_skills`

- 清除技能元数据缓存和文档缓存
- 重新扫描 skills/ 目录并重建索引
- 用于技能文件更新后的显式刷新（会话内默认不热更新，需显式触发）

### 5.2 增强功能（V2 考虑）

| 编号 | 功能 | 说明 |
|------|------|------|
| F13 | 任务类型自动分类 | 在 UserPromptSubmit 阶段自动判断任务类型并推荐候选技能 |
| F14 | 信任等级管理 | 按来源标记技能的信任等级，影响默认风险评估 |
| F15 | 多环境策略 | 支持 dev/staging/prod 不同环境的策略配置 |

## 6 非功能需求

### 6.1 性能

| 指标 | 要求 |
|------|------|
| Hook 处理延迟 | 单次 Hook（SessionStart/PreToolUse/PostToolUse）处理时间 < 50ms |
| MCP 工具响应 | 元工具（list_skills 等）响应时间 < 100ms |
| 缓存命中率 | 会话内技能元数据缓存命中率 > 95% |
| 内存占用 | 插件常驻内存 < 50MB |

### 6.2 可扩展性

- 核心治理逻辑（`src/tool_governance/core/`）不依赖 Claude Code 特定 API，便于未来适配其他平台
- 策略配置通过 YAML 文件定义，支持用户自定义
- 技能可以由用户自行编写并放入 skills/ 目录

### 6.3 可维护性

- 代码遵循 Python 类型提示规范（PEP 484）
- 关键模块有单元测试覆盖
- 使用 Pydantic 做数据验证，减少运行时类型错误
- 模块间通过明确的接口通信，低耦合

### 6.4 安全

- 技能文件作为不可信输入处理：YAML 使用 `safe_load`，文件大小限制，解析失败不影响其他技能
- 高风险操作（涉及 shell、写入、网络、数据库）默认需要用户确认
- 审计日志不可篡改（append-only SQLite 表）
- 策略配置变更需通过文件修改，不可通过 MCP 工具动态修改

### 6.5 兼容性

- Python 3.10+（原定 3.11+，实测 3.10 兼容，已降低门槛）
- 支持 Windows、macOS、Linux 三端
- 兼容 Claude Code 当前稳定版的插件规范

## 7 交互流程

### 7.1 典型使用流程

```
用户发送任务 ──→ [SessionStart Hook]
                    │
                    ├─ 初始化/恢复会话状态
                    ├─ 回收过期 Grant
                    └─ 注入技能目录摘要 (additionalContext)
                    │
              Claude 看到技能目录
                    │
                    ├─ 调用 list_skills() → 获取技能列表
                    ├─ 调用 read_skill("db-ops") → 获取 SOP
                    ├─ 调用 enable_skill("db-ops") → 申请授权
                    │       │
                    │       ├─ [低风险] → 自动授权 → Grant 创建
                    │       └─ [高风险] → 用户确认 → Grant 创建
                    │
                    ├─ 调用 run_skill_action("db-ops", "query", {...})
                    │       │
                    │       └─ [PreToolUse Hook] 检查 active_tools
                    │           ├─ 在白名单 → allow
                    │           └─ 不在 → deny
                    │
                    └─ [PostToolUse Hook] 记录审计日志
```

### 7.2 授权拒绝流程

```
Claude 尝试调用未授权工具
        │
  [PreToolUse Hook]
        │
  工具不在 active_tools 中
        │
  返回 deny + additionalContext:
  "该工具需要先启用对应技能，请使用 read_skill 和 enable_skill 流程"
        │
  Claude 转入正确的授权流程
```

## 8 约束条件

1. **宿主限制**：V1 仅支持 Claude Code 作为宿主环境
2. **语言约束**：核心逻辑使用 Python 3.11+，依赖 LangChain 框架
3. **网络约束**：MCP Server 使用 stdio 通信，不需要网络端口；HTTP Hook 服务（如使用）仅监听 localhost
4. **存储约束**：使用 `${CLAUDE_PLUGIN_DATA}` 目录存储 SQLite 数据库和配置
5. **不修改 Claude Code 核心**：插件只能通过官方提供的 hooks、MCP、skills 机制扩展行为

## 10 术语表

| 术语 | 定义 |
|------|------|
| **Skill（技能）** | 可复用的 SOP + 执行边界的组合，包含 workflow 描述和 allowed_tools 白名单 |
| **Knowledge Plane（知识平面）** | 技能的知识层面：SOP、适用场景、步骤说明、注意事项 |
| **Execution Plane（执行平面）** | 技能的执行层面：allowed_tools 白名单、授权状态、工具装配 |
| **skills_metadata** | 会话中所有已注册技能的元数据索引 |
| **skills_loaded** | 当前会话中已显式启用的技能集合 |
| **active_tools** | 当前轮次中模型可见的最小工具集合，由 skills_loaded 的 allowed_tools 并集计算 |
| **Grant（授权记录）** | 一次 enable_skill 创建的授权条目，包含范围、TTL、状态等 |
| **元工具** | 治理插件暴露给模型的少量固定工具（list_skills、read_skill 等），是治理入口 |
| **Progressive Disclosure（渐进式披露）** | 不一次性暴露所有能力，而是按需逐步展示 |
