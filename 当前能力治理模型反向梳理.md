# 2. allowed_tools / active_tools / whitelist 的当前关系

## 2.1 `allowed_tools` 在哪里定义？

**多个来源：**

1. **Skill 级别**：`SkillMetadata.allowed_tools: list[str]`（`models/skill.py:39`）
   - 从 SKILL.md frontmatter 解析
   - 当 skill 没有 stages 时使用

2. **Stage 级别**：`StageDefinition.allowed_tools: list[str]`（`models/skill.py:22`）
   - 从 SKILL.md frontmatter 的 `stages` 数组解析
   - 当 skill 有 stages 时，覆盖 skill 级别的 allowed_tools

**读取函数：**
- `ToolRewriter.get_stage_tools(skill_meta, current_stage)` (`core/tool_rewriter.py:118`)
  - 决策逻辑：
    - 无 stages → 返回 `skill.allowed_tools`
    - 有 stages + `current_stage=None` → 返回第一个 stage 的 `allowed_tools`
    - 有 stages + `current_stage` 匹配 → 返回该 stage 的 `allowed_tools`
    - 有 stages + `current_stage` 不匹配 → 返回 `[]`（工具全部丢失）

## 2.2 `active_tools` 在哪里生成？

**新路径（权威）：**
- `build_runtime_context(state, metadata, blocked_tools, clock)` → `RuntimeContext.active_tools`
  - 位置：`core/runtime_context.py:96-164`
  - 计算公式：`active_tools = META_TOOLS ∪ (⋃ enabled_skills 的 stage_tools) − blocked_tools`
  - 详细步骤：
    1. 初始化 `tools = set(META_TOOLS)`
    2. 遍历 `state.skills_loaded`
    3. 对每个 skill，检查是否有 grant 且未过期（`runtime_context.py:144-150`）
    4. 调用 `ToolRewriter.get_stage_tools(meta, loaded.current_stage)` 获取该 skill 的工具
    5. `tools.update(stage_tools)`
    6. `tools -= blocked`
    7. 返回 `tuple(sorted(tools))`

**Legacy 路径（已废弃）：**
- `ToolRewriter.recompute_active_tools(state, indexer)` → `state.active_tools`
  - 位置：`core/tool_rewriter.py:59-115`
  - 状态：标记为 DEPRECATED，发出 DeprecationWarning
  - 实现：内部调用 `build_runtime_context` 然后写回 `state.active_tools`

**新路径辅助函数：**
- `compute_active_tools(ctx: RuntimeContext)` → `list[str]`
  - 位置：`core/tool_rewriter.py:152-167`
  - 实现：`return list(ctx.active_tools)`（纯 passthrough）

## 2.3 `whitelist` 在代码中是否是真实变量名？

**否。** `whitelist` 不是任何真实变量名。

**实际情况：**
- `tool_not_available` 是一个**字符串字面量**，用于错误分类
- 出现位置：`hook_handler.py:_classify_deny_bucket()` (225-272行)
- 返回值：`"tool_not_available"` 或 `"wrong_skill_tool"`

**`tool_not_available` 的语义：**
PreToolUse deny 时的错误桶分类，表示"工具不在 active_tools 中"，包括以下情况：
1. 没有任何 skill 被启用
2. 工具不属于任何已索引的 skill
3. 工具属于已启用的 skill，但被 blocked_tools 或 stage 过滤掉了

## 2.4 当前代码里的链路

**完整链路：**

```
SKILL.md (allowed_tools)
    ↓ [SkillIndexer.parse_skill_metadata]
SkillMetadata.allowed_tools / StageDefinition.allowed_tools
    ↓ [enable_skill → state.skills_loaded + active_grants]
SessionState.skills_loaded + active_grants
    ↓ [build_runtime_context]
RuntimeContext.active_tools  ← 权威计算结果
    ↓ [state.sync_from_runtime] (compat shim)
SessionState.active_tools  ← 内存镜像（不持久化）
    ↓ [handle_pre_tool_use]
PreToolUse check: tool_name in ctx.active_tools_set()
```

**关键函数链：**
1. `hook_handler._build_runtime_ctx(rt, state)` → `RuntimeContext`
2. `build_runtime_context(state, metadata, blocked_tools)` → `RuntimeContext`
3. `handle_pre_tool_use(input_data)`:
   - `ctx = _build_runtime_ctx(rt, state)`
   - `if short_name in ctx.active_tools_set(): allow`
   - `else: deny + _classify_deny_bucket(tool_name, ctx)`

## 2.5 `tool_not_available` 的当前含义

**准确定义：**
"工具不在 `active_tools` 中，且不属于 `wrong_skill_tool` 类别"

**判断逻辑（`hook_handler.py:225-272`）：**
1. 如果没有任何 enabled skill → `tool_not_available`
2. 如果工具在某个 enabled skill 的 allowed_tools 中 → `tool_not_available`（说明被 blocked_tools 过滤了）
3. 如果工具在某个 **未启用** skill 的 allowed_tools 中 → `wrong_skill_tool`
4. 其他情况 → `tool_not_available`

**命名问题：**
- 名字暗示存在 `whitelist` 变量，但实际不存在
- 实际语义是"不在 active_tools"，而非"不在 whitelist"
- 与 `blocked_tools` 的关系不清晰（被 blocked 的工具也返回 `tool_not_available`）

# 3. Skill / Stage / Tool 的当前模型

## 3.1 Skill 的表示

**类/数据结构：**
- `SkillMetadata` (pydantic BaseModel, `models/skill.py:25-48`)
  - `skill_id: str`
  - `name: str`
  - `description: str`
  - `risk_level: Literal["low", "medium", "high"]`
  - `allowed_tools: list[str]`
  - `allowed_ops: list[str]`
  - `stages: list[StageDefinition]`
  - `default_ttl: int`
  - `source_path: str`
  - `version: str`

- `SkillContent` (pydantic BaseModel, `models/skill.py:51-61`)
  - `metadata: SkillMetadata`
  - `sop: str` ← SKILL.md body
  - `examples: list[str]`

**SOP 来源：**
- `SkillIndexer.read_skill(skill_id)` → `SkillContent`
- 读取 `SKILL.md` 文件，frontmatter → metadata，body → sop
- 缓存在 `VersionedTTLCache` (TTL=300s)

**运行时进入：**
- `read_skill` MCP tool 返回完整 `SkillContent`
- `PromptComposer` 可能注入 SOP 到 context（未在本次分析范围）

## 3.2 Stage 的表示

**类/数据结构：**
- `StageDefinition` (pydantic BaseModel, `models/skill.py:10-22`)
  - `stage_id: str`
  - `description: str`
  - `allowed_tools: list[str]`

**当前 stage 存储：**
- `LoadedSkillInfo.current_stage: str | None` (`models/state.py:30`)
- 持久化在 `SessionState.skills_loaded[skill_id].current_stage`

**Stage 切换：**
- `change_stage(skill_id, stage_id)` MCP tool (`mcp_server.py:309-350`)
- 验证：skill 已启用 → metadata 存在 → stages 非空 → stage_id 有效
- 写入：`state.skills_loaded[skill_id].current_stage = stage_id`
- 重新计算：`ctx = _build_runtime_ctx(rt, state)` → 新的 `active_tools`

**Stage 对 active_tools 的影响：**
- `ToolRewriter.get_stage_tools(meta, current_stage)`:
  - 无 stages → 返回 `meta.allowed_tools`
  - 有 stages + `current_stage=None` → 返回第一个 stage 的 `allowed_tools`
  - 有 stages + `current_stage` 匹配 → 返回该 stage 的 `allowed_tools`
  - 有 stages + `current_stage` 不匹配 → 返回 `[]`（工具全部丢失）

**allowed_next_stages：**
- **不存在**。当前代码没有 `allowed_next_stages` 字段或验证逻辑。

## 3.3 Tool 的表示

**MCP tool name 格式：**
- Meta-tools: `mcp__tool-governance__list_skills` (完整 MCP 名称)
- Skill tools: `yuque_search` (短名称，由 MCP server 提供)

**命名空间：**
- `META_TOOLS` 使用完整 MCP 名称 (`core/tool_rewriter.py:29-40`)
- `allowed_tools` 使用短名称（从 SKILL.md 配置）
- `active_tools` 混合：meta-tools 用完整名，skill tools 用短名
- PreToolUse 检查时提取短名：`_extract_tool_short_name(tool_name)` (`hook_handler.py:67-74`)

**Tool registry / metadata：**
- **不存在**。当前没有工具级别的 metadata 或 registry。
- 工具仅作为字符串存在于 `allowed_tools` 和 `active_tools` 中。

## 3.4 Skill 是 SOP 还是工具分组？

**当前实现：两者兼具，但偏向工具分组。**

**工具分组特征（主导）：**
- `allowed_tools` 是核心字段，决定 skill 启用后哪些工具可用
- `enable_skill` 的主要效果是扩展 `active_tools`
- Stage 机制用于动态调整工具集合

**SOP 特征（辅助）：**
- `sop` 字段存储操作流程文档
- `read_skill` 可以读取 SOP
- `PromptComposer` 可能注入 SOP（未验证）

**Example 差异：**
- `01-knowledge-link/yuque-knowledge-link`: 只读 skill，`allowed_tools` = 3个查询工具，SOP 占位符
- `02-doc-edit-staged/yuque-doc-edit`: 分阶段 skill，2个 stages，每个 stage 不同 `allowed_tools`，SOP 占位符

# 4. Authorization / Policy / Grant 当前链路

## 4.1 用户/Agent 如何发现 skill

**MCP Tools：**
1. `list_skills()` (`mcp_server.py:74-97`)
   - 返回：`[{skill_id, name, description, risk_level, is_enabled}, ...]`
   - 数据源：`ctx.all_skills_metadata`（从 indexer）

2. `read_skill(skill_id)` (`mcp_server.py:101-125`)
   - 返回：`SkillContent.model_dump()` (metadata + sop + examples)
   - 数据源：`rt.indexer.read_skill(skill_id)`

**Skill Indexer：**
- `SkillIndexer` (`core/skill_indexer.py`)
- 扫描 `skills_dir` 目录，解析所有 `SKILL.md`
- 缓存在 `VersionedTTLCache` (maxsize=100, ttl=300s)
- `current_index()` 返回 `dict[skill_id, SkillMetadata]`

## 4.2 如何启用 skill

**MCP Tool：**
`enable_skill(skill_id, reason="", scope="session", ttl=3600)` (`mcp_server.py:129-194`)

**流程（4-step pattern）：**

**Step 1: Load persisted state**
```python
state = rt.state_manager.load_or_init(sid)
```

**Step 2: Derive runtime context**
```python
ctx = _build_runtime_ctx(rt, state)
meta = ctx.all_skills_metadata.get(skill_id)
```

**Step 3: Mutate persisted-only fields**
```python
# 3.1 Policy evaluation
decision = rt.policy_engine.evaluate(skill_id, meta, state, reason)
if not decision.allowed: return deny

# 3.2 Create grant
capped_ttl = rt.policy_engine.cap_ttl(skill_id, ttl)
grant = rt.grant_manager.create_grant(...)
state.active_grants[skill_id] = grant

# 3.3 Add to skills_loaded
rt.state_manager.add_to_skills_loaded(state, skill_id, version)

# 3.4 Rebuild runtime context
ctx = _build_runtime_ctx(rt, state)
state.sync_from_runtime(ctx.active_tools)  # compat shim
```

**Step 4: Save persisted state**
```python
rt.state_manager.save(state)
rt.store.append_audit(...)
```

**Grant 写入位置：**
- `SessionState.active_grants[skill_id] = Grant(...)`
- 持久化到 SQLite `sessions` 表的 `state_json` 字段

**TTL / expires_at：**
- `Grant.ttl_seconds: int`
- `Grant.expires_at: datetime | None` (计算：`created_at + ttl_seconds`)
- 检查：`build_runtime_context` 过滤过期 grant (`runtime_context.py:148-150`)

**Policy 参与：**
- `PolicyEngine.evaluate(skill_id, meta, state, reason)` → `PolicyDecision`
  - 检查顺序：
    1. `skill_id in policy.blocked_tools` → deny
    2. `skill_policies[skill_id]` 存在 → 检查 `approval_required` / `require_reason` / `auto_grant`
    3. `default_risk_thresholds[meta.risk_level]` → "auto" / "reason" / "approval"
  - `require_reason=True` 且 `reason` 为空 → `decision="reason_required", allowed=False`
  - `approval_required=True` → `decision="approval_required", allowed=False`（未实现实际审批流程）

## 4.3 如何计算当前可用工具

**完整链路：**

```
enabled grants (SessionState.active_grants)
    ↓
build_runtime_context(state, metadata, blocked_tools)
    ↓ [遍历 skills_loaded]
    ↓ [检查 grant 存在且未过期]
    ↓ [ToolRewriter.get_stage_tools(meta, current_stage)]
    ↓ [tools.update(stage_tools)]
    ↓ [tools -= blocked]
RuntimeContext.active_tools
```

**Stage 过滤：**
- `current_stage` 从 `LoadedSkillInfo.current_stage` 读取
- `get_stage_tools` 根据 stage 返回对应 `allowed_tools`

**Blocked_tools 过滤：**
- `blocked_tools` 从 `GovernancePolicy.blocked_tools` 读取
- 在 `build_runtime_context` 最后一步：`tools -= blocked` (`runtime_context.py:156`)

**Grant 过期过滤：**
```python
grant = state.active_grants.get(skill_id)
if grant is None or (grant.expires_at and grant.expires_at < clock):
    continue  # 跳过该 skill，不贡献工具
```

## 4.4 PreToolUse 如何判断 allow / deny

**入口：**
`handle_pre_tool_use(input_data)` (`hook_handler.py:275-355`)

**输入：**
- `input_data: dict` (包含 `tool_name`, `session_id`)

**读取 state / context：**
```python
session_id = discover_session_id(input_data)
tool_name = input_data.get("tool_name", "")
short_name = _extract_tool_short_name(tool_name)

# Fast-path: meta-tools 无需检查
if short_name in _META_SHORT_NAMES or tool_name in META_TOOLS:
    return allow

# Load + derive
state = rt.state_manager.load_or_init(session_id)
ctx = _build_runtime_ctx(rt, state)
```

**检查顺序：**
1. Meta-tools → allow（无条件）
2. `short_name in ctx.active_tools_set()` 或 `tool_name in ctx.active_tools_set()` → allow
3. 否则 → deny + `_classify_deny_bucket(tool_name, ctx)`

**返回结构：**
```python
# Allow
{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "allow"
  }
}

# Deny
{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "deny",
    "permissionDecisionReason": "Tool '...' is not in active_tools. ...",
    "additionalContext": "To use this tool, first discover ..."
  }
}
```

**错误类型：**
- `tool_not_available`: 工具不在 active_tools，且不属于未启用 skill
- `wrong_skill_tool`: 工具属于某个未启用的 skill
- `parameter_error`: PostToolUse 检测到工具返回错误（不在 PreToolUse）

# 5. blocked_tools 的当前模型

## 5.1 配置位置

**YAML 配置：**
- `config/default_policy.yaml:16` → `blocked_tools: []`
- `examples/02-doc-edit-staged/config/demo_policy.yaml:16-17` → `blocked_tools: [run_command]`

**数据模型：**
- `GovernancePolicy.blocked_tools: list[str]` (`models/policy.py:51`)

## 5.2 级别

**全局 policy 级别。**

不存在 skill-level 或 stage-level 的 `blocked_tools`。

## 5.3 检查时机

**唯一检查点：**
`build_runtime_context` 最后一步 (`runtime_context.py:156`)：
```python
tools -= blocked
```

**不在以下位置检查：**
- PreToolUse（直接检查 `active_tools`，已经过滤了 blocked）
- PolicyEngine（只检查 `skill_id in blocked_tools`，用于 skill 级别拒绝）

## 5.4 与 active_tools / allowed_tools 的检查顺序

**计算顺序（`build_runtime_context`）：**
1. `tools = set(META_TOOLS)`
2. 遍历 enabled skills，`tools.update(stage_tools)`
3. `tools -= blocked`
4. `return tuple(sorted(tools))`

**结果：**
- `active_tools = (META_TOOLS ∪ allowed_tools) − blocked_tools`
- `blocked_tools` 优先级最高，即使在 `allowed_tools` 中也会被移除

## 5.5 同时不在 active_tools 且在 blocked_tools 的情况

**不可能发生。**

因为 `blocked_tools` 在计算 `active_tools` 时已经过滤掉，所以：
- 如果工具在 `blocked_tools` 中，它一定不在 `active_tools` 中
- PreToolUse 只检查 `active_tools`，返回 `tool_not_available`

**错误类型：**
被 blocked 的工具返回 `tool_not_available`，而非专门的 "blocked" 错误。

## 5.6 Tests / Examples 覆盖

**Tests：**
- `test_tool_rewriter.py:96-108`: `test_blocked_tools_removed`
- `test_runtime_context.py:78-86`: `test_empty_state_with_blocked_subtracts_meta`

**Examples：**
- `examples/02-doc-edit-staged/config/demo_policy.yaml`: `blocked_tools: [run_command]`

## 5.7 Demo 中 Bash 返回 tool_not_available 的原因

**原因分析：**

假设 demo 中 `Bash` 工具被 blocked：
1. `blocked_tools: [Bash]` 在 policy.yaml
2. `build_runtime_context` 计算时：`tools -= {"Bash"}`
3. `active_tools` 不包含 `Bash`
4. PreToolUse 检查：`"Bash" not in ctx.active_tools_set()` → deny
5. `_classify_deny_bucket("Bash", ctx)`:
   - `Bash` 不在任何 enabled skill 的 `allowed_tools` 中
   - `Bash` 可能在某个未启用 skill 的 `allowed_tools` 中 → `wrong_skill_tool`
   - 或者不在任何 skill 中 → `tool_not_available`

**结论：**
这是**代码行为**，不是配置问题。`blocked_tools` 不会产生专门的 "blocked" 错误，而是通过 `active_tools` 过滤后返回 `tool_not_available` 或 `wrong_skill_tool`。

## 6.2 持久化字段

**持久化到 SQLite（`state_manager.py:103-106`）：**


**`DERIVED_FIELDS`（`models/state.py:101`）：**
```python
DERIVED_FIELDS: ClassVar[frozenset[str]] = frozenset({"active_tools", "skills_metadata"})
```

**持久化字段：**
- `session_id`
- `skills_loaded`
- `active_grants`
- `created_at`
- `updated_at`

## 6.3 不持久化字段

**Derived 字段（运行时计算）：**

**为什么不持久化：**
- `active_tools`: 可从 `skills_loaded` + `active_grants` + `metadata` + `policy` 重新计算
- `skills_metadata`: 权威来源是 `SkillIndexer`，持久化会导致 stale data

## 6.4 RuntimeContext 包含的字段

**完整字段（`core/runtime_context.py:62-87`）：**
- `active_tools: tuple[str, ...]` ← 计算结果，sorted
- `enabled_skills: tuple[EnabledSkillView, ...]` ← 已启用且有 metadata 的 skill
- `all_skills_metadata: Mapping[str, SkillMetadata]` ← 完整 skill 索引
- `policy: PolicySnapshot` ← 包含 `blocked_tools`
- `clock: datetime` ← 构建时间戳

**特性：**
- `frozen=True` dataclass，不可变
- 不持久化，每个 turn 重新构建
- 纯函数构建，无副作用

## 6.5 `build_runtime_context()` 如何构建

**函数签名（`core/runtime_context.py:96-101`）：**
```python
def build_runtime_context(
    state: SessionState,
    metadata: Mapping[str, SkillMetadata] | None = None,
    blocked_tools: Iterable[str] = (),
    clock: datetime | None = None,
) -> RuntimeContext:
```

**输入来源：**
1. `state`: 从 `StateManager.load_or_init(session_id)` 加载
2. `metadata`: 从 `SkillIndexer.current_index()` 或 fallback 到 `state.skills_metadata`
3. `blocked_tools`: 从 `GovernancePolicy.blocked_tools` 或 `ToolRewriter.blocked_tools`
4. `clock`: 默认 `datetime.utcnow()`

**构建步骤：**
1. 初始化 `tools = set(META_TOOLS)`
2. 遍历 `state.skills_loaded`:
   - 检查 metadata 存在
   - 检查 grant 存在且未过期
   - 获取 stage_tools
   - `tools.update(stage_tools)`
3. `tools -= blocked`
4. 构建 `EnabledSkillView` 列表
5. 返回 frozen `RuntimeContext`

## 6.6 `active_tools` 应该属于哪里？

**当前状态：**
- **权威来源**：`RuntimeContext.active_tools`（每个 turn 计算）
- **内存镜像**：`SessionState.active_tools`（通过 `sync_from_runtime()` 同步）
- **不持久化**：`DERIVED_FIELDS` 排除

**设计意图：**
- `RuntimeContext.active_tools` 是唯一真相来源
- `SessionState.active_tools` 是 compat shim，供未迁移的代码读取
- 最终目标：所有代码迁移到 `RuntimeContext`，移除 `SessionState.active_tools`

## 6.7 Legacy fallback

**存在的 fallback：**

1. **Metadata fallback（`runtime_context.py:128-130`）：**
```python
effective_metadata: Mapping[str, SkillMetadata] = (
    metadata if metadata else state.skills_metadata

```
- 如果 `metadata=None`，fallback 到 `state.skills_metadata`
- 用于冷启动场景（indexer 未初始化）

2. **`recompute_active_tools` adapter（`tool_rewriter.py:59-115`）：**
- 标记为 DEPRECATED
- 内部调用 `build_runtime_context`
- 写回 `state.active_tools`

## 6.8 哪些地方仍使用旧路径？

**已迁移到 RuntimeContext：**
- `hook_handler.py`: 所有 hook 入口点

## 7.3 active_tools computation flow

```
SessionState (持久化状态)
├─ skills_loaded: dict[skill_id, LoadedSkillInfo]
│   └─ current_stage: str | None
└─ active_grants: dict[skill_id, Grant]
    └─ expires_at: datetime | None

SkillIndexer.current_index()
└─ dict[skill_id, SkillMetadata]
    ├─ allowed_tools: list[str]
    └─ stages: list[StageDefinition]
        └─ allowed_tools: list[str]

GovernancePolicy
└─ blocked_tools: list[str]

    ↓ [传入 build_runtime_context]

build_runtime_context(state, metadata, blocked_tools, clock):
    tools = set(META_TOOLS)
    
    for skill_id, loaded in state.skills_loaded.items():
        meta = metadata.get(skill_id)
        if meta is None: continue  # 安全降级
        
        grant = state.active_grants.get(skill_id)
        if grant is None: continue  # 无授权
        if grant.expires_at and grant.expires_at < clock:
            continue  # 已过期
        
        stage_tools = ToolRewriter.get_stage_tools(meta, loaded.current_stage)
        tools.update(stage_tools)
    
    tools -= blocked_tools
    
    return RuntimeContext(
        active_tools=tuple(sorted(tools)),
        enabled_skills=...,
        all_skills_metadata=metadata,
        policy=PolicySnapshot(blocked_tools=blocked_tools),
        clock=clock
    )

    ↓

RuntimeContext.active_tools: tuple[str, ...]
```

**关键函数：**
- `build_runtime_context(state, metadata, blocked_tools, clock)`
- `ToolRewriter.get_stage_tools(skill_meta, current_stage)`

```
Claude 尝试调用工具
    ↓ [Claude Code hook 机制]

    ↓

    ↓

    bucket = _classify_deny_bucket(tool_name, ctx)
    SQLiteStore.append_audit(session_id, "tool.call", decision="deny", 
                             detail={"error_bucket": bucket})

        "permissionDecisionReason": "Tool '...' is not in active_tools. ...",
        "additionalContext": "To use this tool, first discover ..."

```

**_classify_deny_bucket 逻辑：**
```
_classify_deny_bucket(tool_name, ctx):
    short_name = _extract_tool_short_name(tool_name)
    enabled_ids = ctx.enabled_skill_ids()
    
    if not enabled_ids:
        return "tool_not_available"  # 没有任何 skill 启用
    
    # 检查是否在已启用 skill 的 allowed_tools 中
    for sv in ctx.enabled_skills:
        if short_name in sv.metadata.allowed_tools:
            return "tool_not_available"  # 在 allowed_tools 但被过滤了
        for stage in sv.metadata.stages:
            if short_name in stage.allowed_tools:
                return "tool_not_available"
    
    # 检查是否在未启用 skill 的 allowed_tools 中
    for skill_id, meta in ctx.all_skills_metadata.items():
        if skill_id in enabled_ids: continue
        if short_name in meta.allowed_tools:
            return "wrong_skill_tool"  # 属于未启用的 skill
        for stage in meta.stages:
            if short_name in stage.allowed_tools:
                return "wrong_skill_tool"
    
    return "tool_not_available"  # 其他情况
```

**关键函数：**
- `hook_handler.handle_pre_tool_use(input_data)`
- `hook_handler._extract_tool_short_name(tool_name)`
- `hook_handler._classify_deny_bucket(tool_name, ctx)`

## 重点命名风险

### 1. `allowed_tools` vs `active_tools` 混淆

**问题：**
- `allowed_tools` 是声明（输入），`active_tools` 是计算结果（输出）
- 两者关系：`active_tools = f(allowed_tools, grants, stage, blocked_tools)`
- 开发者容易混淆"声明允许"和"实际可用"

**建议：**
- 重命名 `active_tools` → `available_tools` 或 `authorized_tools`
- 或重命名 `allowed_tools` → `declared_tools` 或 `skill_tools`

### 2. `tool_not_available` 误导

**问题：**
- 名字暗示存在 `whitelist` 变量，但实际不存在
- 与 `blocked_tools` 的关系不清晰
- 被 blocked 的工具也返回 `tool_not_available`，而非 "blocked"

**建议：**
- 重命名为 `tool_not_available` 或 `unauthorized_tool`
- 或引入新的错误类型 `tool_blocked`，与 `tool_not_available` 区分

### 3. `enabled_skills` vs `skills_loaded` 重复

**问题：**
- `SessionState.skills_loaded` 和 `RuntimeContext.enabled_skills` 语义相同
- 两个名字造成混淆

**建议：**
- 统一命名为 `enabled_skills`
- `SessionState.enabled_skills: dict[str, LoadedSkillInfo]`
- `RuntimeContext.enabled_skills: tuple[EnabledSkillView, ...]`

### 4. `SessionState.active_tools` 和 `skills_metadata` 应移除

**问题：**
- 两者都是 derived fields，不持久化
- 存在于 `SessionState` 中造成"持久化状态"的误解
- `sync_from_runtime()` 是 compat shim，增加复杂度

**建议：**
- 移除 `SessionState.active_tools` 和 `SessionState.skills_metadata`
- 迁移所有读取者到 `RuntimeContext`
- 移除 `sync_from_runtime()` 方法

### 5. `stage.allowed_tools` 覆盖语义不清晰

**问题：**
- 当 skill 有 stages 时，`stage.allowed_tools` 完全覆盖 `skill.allowed_tools`
- 代码中没有明确文档说明这是"覆盖"而非"合并"

**建议：**
- 在 `StageDefinition` 和 `SkillMetadata` 的 docstring 中明确说明
- 或考虑引入 `stage.additional_tools` 表示"增量"语义

# 9. 最终结论

## 9.1 当前模型一句话总结

**Tool-Gate 当前实现了一个基于 Skill 的工具授权模型：**

Skill 声明 `allowed_tools`（可选分 stage），用户通过 `enable_skill` 创建 Grant，系统从 enabled skills 的 `allowed_tools` 计算出 `active_tools`（经过 grant 过期检查、stage 过滤、blocked_tools 过滤），PreToolUse 检查工具是否在 `active_tools` 中来决定 allow/deny。

## 9.2 allowed_tools / active_tools / whitelist 当前最准确的关系

**三者关系：**

1. **`allowed_tools`（输入）**：
   - Skill/Stage 级别的工具声明
   - 来源：SKILL.md frontmatter
   - 语义：该 skill 在该 stage "声称可以使用" 的工具

2. **`active_tools`（输出）**：
   - Turn 级别的工具集合
   - 来源：`build_runtime_context` 计算
   - 语义：当前 turn "实际可用" 的工具
   - 计算公式：`active_tools = META_TOOLS ∪ (⋃ enabled_skills 的 stage_tools) − blocked_tools`
   - 过滤条件：
     - Skill 必须 enabled（在 `skills_loaded` 中）
     - Grant 必须存在且未过期（在 `active_grants` 中）
     - 根据 `current_stage` 选择对应的 `allowed_tools`
     - 减去 `blocked_tools`

3. **`whitelist`（不存在）**：
   - 仅作为错误名 `tool_not_available` 出现
   - 实际语义：工具不在 `active_tools` 中
   - 命名误导：暗示存在 whitelist 变量，但实际不存在

**准确表述：**
- `allowed_tools` 是 skill 的"工具声明"
- `active_tools` 是 turn 的"可用工具集合"
- `whitelist` 不存在，`tool_not_available` 实际是 "not in active_tools"

## 9.3 如果未来重构，最应该先统一哪几个概念名

**优先级排序：**

### 1. 最高优先级：`active_tools` → `available_tools`

**理由：**
- `active_tools` 是核心概念，出现在所有关键路径
- "active" 语义不准确，容易误解
- 重命名影响面大，但收益最高

**影响范围：**
- `RuntimeContext.active_tools`
- `SessionState.active_tools`（应同时移除）
- `build_runtime_context` 返回值
- `handle_pre_tool_use` 检查逻辑
- 所有测试和文档

### 2. 高优先级：`tool_not_available` → `tool_not_available`

**理由：**
- 误导性最强的命名
- 暗示不存在的 `whitelist` 概念
- 与 `blocked_tools` 关系不清晰

**影响范围：**
- `_classify_deny_bucket` 返回值
- Audit log 记录
- 错误消息

### 3. 中优先级：统一 `enabled_skills` / `skills_loaded`

**理由：**
- 两个名字表示同一概念
- 造成代码阅读困难

**建议：**
- `SessionState.skills_loaded` → `SessionState.enabled_skills`
- 保持 `RuntimeContext.enabled_skills` 不变

### 4. 中优先级：移除 `SessionState.active_tools` 和 `skills_metadata`

**理由：**
- Derived fields 不应存在于持久化状态中
- `sync_from_runtime()` 是技术债

**步骤：**
1. 迁移 `policy_engine.is_tool_allowed` 到接受 `RuntimeContext`
2. 移除 `sync_from_runtime()` 调用
3. 移除 `SessionState.active_tools` 和 `skills_metadata` 字段
4. 更新 `DERIVED_FIELDS`（应为空）

### 5. 低优先级：`allowed_tools` → `declared_tools`

**理由：**
- 虽然有混淆风险，但 `allowed_tools` 已广泛使用
- 重命名成本高，收益相对较低

**建议：**
- 保持 `allowed_tools` 不变
- 在文档中明确说明与 `available_tools` 的区别
